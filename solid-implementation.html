Cursor bundle dissection and Multi migration plan

Solid implementation notes

Drafted on 2026-06-07 from Cursor's bundled workbench, refreshed anomalyco/opencode, and the current Multi app source.

Cursor Solid 1.9.7 Cursor Agents Window is React-rooted opencode is the full-Solid reference Multi full migration: very high risk

Executive judgment

Full migration difficulty: very high. A no-React Solid migration for Multi is an app rewrite, not a component swap. The hard blockers are Multikit/Base UI, the composer prompt editor, markdown rendering, virtual timeline behavior, routing/query/hotkey hook layers, and React-specific icon/component exports.

Cursor does not appear to prove that its whole agent window was migrated to Solid. The newer Agents Window is bootstrapped by a React root. Cursor uses Solid as a workbench island runtime, a service bridge, settings/editor widgets, and a bridge that lets Solid mount React components.

opencode is the better reference for a real full-Solid agent app: a Vite Solid SPA, shallow routing, provider scopes, TanStack Solid Query, SSE coalescing, explicit root lifetimes, keyed timeline rows, and imperative terminal integration under Solid lifecycle.

Runtime shape

  • Q_(component, element, instantiationService, options) creates a full-size host div, renders a Solid component into it, injects a service context, and returns dispose() and focus().
  • Ir()/rnt() read the injected service context and lazily resolve allowed VS Code services through instantiationService. The allowed-service set and cache are stored on symbols in the context object.
  • Vw(), AV(), ok() bridge Cursor events, observables, and settings into Solid signals with cleanup guards.
  • Qy()/VF create a Solid wrapper around React components. VF calls ReactDOM.createRoot, renders with createElement, updates on Solid effects, and unmounts on cleanup.

How Cursor's Agents Window is made

  1. Glass window constants define the product surface: Teh = "Agents Window", cursor.openOrFocusGlassWindow, and workbench.agentsWindowButton.enabled.
  2. B8k.open() prepares workspace services, creates a hidden workspaces-container, creates the root workspace, builds agent repository proxies, then calls XEI({ container: document.body, ... }).
  3. XEI() marks the body as Glass mode, creates a scoped context key service, instantiates the Glass theme service, optionally registers a shared React bridge controller, then calls ReactDOM.createRoot(container).
  4. XEI() renders xxI, the compiled Glass root component. That root wires authentication, clipboard, command services, a portal root, the hidden workspace container, alert dialog provider, service provider, and theme provider.
  5. The root state layer selects and loads agents through Glass active-agent services, agent repository services, and composer data handles. A selected agent resolves to a composer-backed agent reference.
  6. Transcript and prompt components come from the React package chunk. Solid can wrap React components or own small islands, but the root and transcript stack remain React.

It uses React 19, React Compiler, TanStack React Router/Query/Virtual/Hotkeys, @effect/atom-react, @lexical/react, @tiptap/react, streamdown, React Multikit, central-icons React exports, and Zustand.

Full migration plan

This plan assumes a true full migration: no React runtime in the final app agent surface. A Cursor-style mixed runtime is much easier, but that is not a full migration.

  1. Foundation: add Solid app root, Vite Solid plugin path, Solid Router, Solid Query, and a service/provider shell. Keep React root untouched until the Solid route can boot behind a dev flag.
  2. Design system: create Solid Multikit primitives for button, menu, popover, command, dialog, tooltip, tabs, scroll area, and sidebar. Keep tokens and Tailwind utilities. Replace React central-icons imports with Solid-compatible wrappers or generated components.
  3. State adapters: expose vanilla Zustand subscriptions to Solid, build Solid wrappers for server state atoms, and move query options to framework-neutral files consumed by Solid Query.
  4. Composer editor decision: either implement a Solid-compatible editor surface over Lexical core APIs or replace the editor with a ProseMirror/Tiptap path that can own mentions, chips, slash commands, image attachments, and selection geometry without React DecoratorNodes.
  5. Composer shell: port ComposerInput after the editor and primitives exist. Preserve configurable keybindings, pending approval panels, queues, context usage bars, subagent tray, attachments, and inline-edit reuse.
  6. Command menu: port the live virtual anchor and fixed top placement contract. The stale AGENTS path now maps to components/chat/composer/command-menu/menu.tsx and anchor.ts.
  7. Timeline: port row derivation first, then Solid virtualizer rendering. Copy opencode's bias toward stable keys, keyed row mapping, measurement cache, and manual bottom anchoring.
  8. Markdown: choose a non-React streaming markdown renderer or isolate a temporary compatibility layer only during migration. Final full migration needs no React streamdown.
  9. Shell and sidebar: port agent sidebar, shell host, terminal drawer, and workbench panels after composer and timeline are usable.
  10. Verification: run pnpm run typecheck, then targeted tests for any changed test files. Add focused coverage for editor serialization, command menu placement, streaming timeline rows, pending approvals, queues, and keybindings.

Effort estimate

Scope Rough effort Notes
Cursor-style Solid islands 4 to 8 engineer-weeks Keep React app and React Multikit. Add Solid host islands where useful.
Agent surface full Solid migration 14 to 24 engineer-weeks Composer, prompt editor, timeline, markdown, shell/sidebar, primitives.
Whole app full Solid migration 20 to 35+ engineer-weeks Includes routing/query/devtools/settings/Multikit coverage and regression hardening.
Risk driver: the prompt editor and Multikit are on the critical path. If either remains React-only, the result is a mixed runtime, not a full migration.

Recommendation

Use Cursor for the bridge pattern, not as proof of a completed Solid agent migration. Use opencode as the full-Solid implementation reference.

If the goal is lower risk and product momentum, build a Solid island host first and port isolated tool surfaces. If the goal is a hard full migration, start by porting Multikit and the prompt editor architecture before moving timeline or shell code. Without those two foundations, the migration will accumulate adapters that are harder to delete than the original React code.
