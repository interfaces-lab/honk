Pi SDK + MCP Research Plan for Honk

Lower-level comparison of Craft Agents’ MCP bridge, Pi SDK internals, and Honk’s current runtime layer, followed by a concrete implementation plan for adding MCP as a first-class Honk runtime capability.

ResearchMCPPi SDKHonk Runtime

Executive summary

Craft did not use a native Pi MCP API. Pi intentionally has no built-in MCP layer. Craft handled MCP by converting discovered MCP tools into ordinary Pi ToolDefinition custom tools, then proxying execution back to a centralized MCP client pool.

Honk currently embeds Pi directly in-process through ThreadAgentRuntime, which is simpler than Craft’s subprocess model. Honk can implement MCP without Craft’s JSONL bridge by adding an in-process MCP manager and registering dynamic Pi tools through Honk-owned extensions or session custom tools.

Core idea

One centralized McpClientPool owns real MCP connections. Backends receive proxy tool definitions and route tool execution through the pool.

mcp__{sourceSlug}__{originalToolName}

Craft execution flow

LLM
  -> Pi tool call: mcp__linear__searchIssues
  -> pi-agent-server custom ToolDefinition.execute()
  -> JSONL tool_execute_request
  -> PiAgent main process
  -> McpClientPool.callTool()
  -> official MCP Client.callTool()
  -> result text
  -> JSONL tool_execute_response
  -> Pi tool result message
  -> next LLM turn

Craft design details

  • Pool owns connection lifecycle: HTTP/Streamable MCP, stdio MCP, and in-process API sources.
  • Tool discovery is cached: listTools() runs at connect/sync time; proxy definitions are derived from cached MCP tool schemas.
  • Pi sees normal tools: Craft passes proxy tools through customTools and includes every name in the tools allowlist.
  • Permissions stay outside Pi: every tool invokes a pre-tool-use JSONL handshake before execution.
  • Dynamic tool changes recreate Pi sessions: Craft marks toolsChanged, disposes the session before the next prompt, then resumes via Pi session storage.
  • Schema patching: Craft adds optional _intent and _displayName properties to strict schemas, then strips them before real execution.

Tool model

Pi tools are plain ToolDefinition objects wrapped into low-level AgentTool instances:

{
  name,
  label,
  description,
  parameters,
  execute(toolCallId, params, signal, onUpdate, ctx)
}

There is no native MCP semantic path. A tool named mcp__github__search is just a normal custom tool unless the embedding app gives it meaning.

Agent loop details

  1. User message is appended.
  2. Provider stream starts through streamSimple().
  3. Assistant deltas emit message_update.
  4. Assistant tool calls are collected.
  5. Tool calls preflight sequentially, then execute in parallel by default.
  6. Tool results become toolResult messages.
  7. The loop continues until no tool calls and no queued follow-ups remain.

Recommended Honk architecture

Preferred path: in-process MCP runtime

Do not copy Craft’s subprocess JSONL bridge unless isolation becomes necessary. Honk can register MCP tools directly into the existing Pi session because Honk already owns Pi in-process.

DesktopRuntimeHost
  -> McpRuntimeRegistry
    -> McpClientConnection(s)
    -> MCP tool definitions
  -> ThreadAgentRuntime
    -> Honk MCP extension pi.registerTool(...)
    -> Pi AgentSession tool registry
    -> tool.execute() calls MCP client directly

New concepts

  • McpRuntimeRegistry: process-level owner for MCP server configs, connections, discovery, and cleanup.
  • McpToolCatalog: normalized list of available tools by server slug.
  • McpPiToolAdapter: converts MCP Tool schemas into Pi ToolDefinitions.
  • McpPermissionPolicy: integrates with Honk’s runtime policy / interaction mode before calling the real MCP server.
  • McpResultNormalizer: converts text/image/audio/resource MCP content into model-safe text, saved files, and UI details.

Implementation plan

Phase 1: Runtime MCP core

  1. Create packages/runtime/src/mcp/client.ts using @modelcontextprotocol/sdk client transports.
  2. Support Streamable HTTP first; add stdio behind an explicit local-execution setting.
  3. Add env filtering for stdio based on Craft’s blocked env list, adapted to Honk credentials.
  4. Create McpRuntimeRegistry with connect, disconnect, listTools, callTool, and dispose.
  5. Normalize server slugs and tool names to mcp__{slug}__{tool}.

Phase 2: Pi tool adapter

  1. Create a converter from MCP JSON schema to Pi-compatible TypeBox/JSON schema.
  2. Strip unsupported schema metadata such as remote $schema URIs.
  3. Generate ToolDefinition objects with promptSnippet so tools are visible in Pi’s system prompt.
  4. Return details with { kind: "mcp", serverSlug, originalToolName } for Honk UI projection.
  5. Throw on MCP error results only when appropriate; otherwise preserve MCP isError semantics in details and returned text.

Phase 3: Dynamic registration through a Honk extension

  1. Add createMcpRuntimeExtension() to Honk’s desktop extension factory list.
  2. On session_start, register all active MCP tools with pi.registerTool().
  3. When MCP tool catalog changes, dynamically register new tools and call pi.setActiveTools() to include them.
  4. Avoid session recreation unless Pi dynamic registration proves insufficient for system prompt refresh.

Phase 4: Permissions and interaction modes

  1. Run MCP calls through Honk policy before execution.
  2. Classify tools as read/write using server annotations when available and name heuristics otherwise.
  3. In Ask mode, default to read-only MCP tools only.
  4. For write-like MCP tools, require the same approval UX as mutating shell/file tools.

Phase 5: Result handling

  1. Normalize MCP content blocks into model-safe text.
  2. Save binary image/audio/resource payloads into the thread/session data directory.
  3. Apply Honk truncation/large-output behavior before returning content to Pi.
  4. Attach structured metadata so timeline rendering can show provider, operation, result size, and saved artifact links.

Phase 6: UI and configuration

  1. Add a minimal settings/config surface for MCP servers.
  2. Expose connected/disconnected/error states in the environment or runtime snapshot.
  3. Update timeline MCP rendering to use structured details instead of only name heuristics.
  4. Optional: add @ or command menu surfaces later, but do not block core runtime on source UX.

Risks

  • Schema compatibility: MCP servers emit broad JSON Schema. Pi tool validation may reject unsupported schema constructs unless normalized.
  • Token bloat: exposing many MCP tools directly can inflate the system prompt. Tool filtering and per-thread activation matter.
  • Stdio security: local MCP servers are arbitrary subprocesses. Gate behind explicit trust and env filtering.
  • Dynamic registration: Pi supports dynamic pi.registerTool(), but system prompt refresh behavior should be verified with characterization tests.
  • Provider compatibility: some models are sensitive to large tool catalogs and complex schemas.

Verification checklist

  • Unit test tool-name mapping: mcp__slug__tool round-trips to original MCP tool.
  • Unit test schema normalization against representative MCP schemas.
  • Integration test HTTP MCP server: list tools and call tool from a Pi session.
  • Integration test stdio MCP server with env filtering.
  • Runtime test dynamic tool addition after session creation.
  • UI projection test structured MCP details render as MCP timeline entries.
  • Security test Ask mode blocks write-like MCP tools.

Non-goals for first pass

  • No Craft-style full source marketplace.
  • No OAuth source setup automation.
  • No cross-backend MCP HTTP proxy server unless Cursor/Codex integration needs it.
  • No broad API-source abstraction; start with real MCP servers.
