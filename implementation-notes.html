Decisions, deviations & tradeoffs

Decision Scope: ship the surgical four, defer the two structural rewrites

The review ranked six fixes. I implemented the four that are each isolated to one file, high-confidence, and directly eliminate the crash + the bulk of the freeze/energy (plus one bonus per-event size cut, "E"), and deferred the two large structural items (off-main-thread execution; UI unification) to their own PRs — they carry real lifecycle/parity risk and deserve focused review. Rationale per item below; deferred items at the bottom.

A · Cap host runtime event arrays desktop-runtime-host.ts

Decision Amortized 2× trim, not trim-on-every-push

A naïve cap (splice one element whenever length > max) is an O(max) array shift on every event — itself a per-event cost under the exact load we're fixing. Instead boundedPush lets the array grow to 2×max then trims back to max in one splice → amortized O(1) per push, steady-state memory bounded at 2×max.

Decision Two different caps for two different roles

Global runtimeEvents capped at MAX_RUNTIME_EVENTS_IN_SNAPSHOT (500): it only ever feeds the host snapshot, which already exposes at most 500. Per-thread runtimeEventsByThreadId capped at MAX_RUNTIME_EVENTS_PER_THREAD (1000): it feeds the live display re-projection, so it must comfortably exceed one in-flight turn's live events.

Risk / tradeoff Full re-projection after a >1000-event uncommitted turn

refreshDisplayTimeline re-projects over the (now capped) per-thread array. If a single turn streamed >1000 live events before any session-tree commit, the full re-projection could miss the earliest live items. In practice this is safe: (1) the incremental projector keeps the current timeline in displayTimelines independently of the event array, and (2) committed work is restored from the session tree, and turns commit per-tool / per-message so the live backlog stays well under the cap. The snapshot-cap test still passes unchanged (the snapshot path still slices to the last 500).

B · Coalesce display-timeline emits desktop-runtime-host.ts

Decision Timer-coalesced trailing flush at 16 ms, per-thread

Each display-timeline event carries the entire projection and was emitted once per runtime event, then full-Effect-encoded in main + structured-cloned to every window + full-Effect-decoded in the renderer. Intermediate ones are redundant (the app renders from the latest timeline; reduceHostEvent upserts by threadId). A dirty-set + setTimeout(16ms) trailing flush emits at most one per thread per frame. Chose a timer over a microtask because pi events arrive asynchronously (per SSE chunk), so a microtask wouldn't coalesce across them; 16 ms ≈ 60 fps matches Cursor's render cap.

Changed behavior display-timeline emission is now async (≤16 ms)

Previously synchronous per event. No test asserted synchronous emit timing, and getHostSnapshot() still reads displayTimelines directly (kept fresh by the per-event incremental projection), so snapshot reads are unaffected. Only the push is coalesced; the expensive encode/broadcast/decode is what we cut. Flush timer is cleared in dispose(); a thread is removed from the dirty set in disposeRuntime.

Decision Did NOT coalesce runtime-event

Each runtime-event is semantically distinct (unique id; subagent activity sync and ingestion both ride on it), so dropping/merging them is unsafe without the larger delta-refactor. Fix C shrinks the subagent ones at the source and fix E halves all of them instead.

C · Throttle + bound subagent publishes subagent-extension.ts

Decision onUpdate(toolDetails) → zero-arg notify()

The core source fix. runSubagentTask previously computed toolDetails(state) — a full copy of the combined activities array across all concurrent children — and published it on every child SSE chunk. It now calls a zero-arg notify(); a leading+trailing throttle computes toolDetails and publishes only when it actually fires (≤10 Hz). The full-array copy no longer runs per chunk.

Decision 100 ms interval, flush-on-completion, unthrottled final result

100 ms (10 Hz) sits between Cursor's 16 ms render cap and pi-subagents' 250 ms poll — live enough, not a flood. flush() on completion guarantees the tray lands on the terminal state, and the returned tool result is computed unthrottled, so throttling can never drop data — at worst it skips an intermediate progress frame.

Tradeoff Clamp progress detail (head for text, tail for output)

A single child tool result can be ~50 KB and was mirrored into the parent activity and re-shipped. Detail is now clamped: assistant/reasoning keep their head (16 KB — the message as written); command/tool output keeps its tail (8 KB — the most recent output). run.finalText / lastAssistantText stay full — only the displayed detail is clamped, and the child's complete output remains in its own session. Tradeoff: the live tray may show a truncated preview for very large outputs; the full final answer still reaches the parent conversation as the tool result.

D · Bound ingestion dedup sets runtime-ingestion.ts

Decision FIFO eviction at 5000 keys

The two module-singleton Sets grew for the whole app lifetime. rememberPersistedKey evicts the oldest once over 5000. Tradeoff a re-arriving evicted (very old) key could be re-dispatched — harmless because backend persistence is keyed by commandId (idempotent).

E · Drop the unused raw field event-projection.ts

Decision Stop populating event.raw

raw: event duplicated the whole pi event already shallow-copied into data — for a message_update that's the full accumulated message a third time (it appears as data.message, data.assistantMessageEvent.partial, and raw). Grep confirmed zero consumers of event.raw across app/runtime/desktop. Dropping it ~halves every runtime-event payload that crosses IPC.

Decision I kept raw in the contract schema as Schema.optional rather than removing it — removing a schema field is a broader change and decoders already tolerate its absence; the producer simply no longer sets it. (Noted as an intentionally dead-but-optional schema field.) I did not attempt to also slim data for message events: many payload.data readers live in the 3700-line session-logic.ts, so that belongs in the delta-refactor (deferred).

Incidental Removed two pre-existing meaningless void operators

void this.dispatchCommand(...) (×2) in runtime-ingestion.ts — the method returns void, so the operator was meaningless and oxlint flagged it once the file was in my diff. Pre-existing (not introduced here); cleaned up to keep the edited file lint-clean.

Phase 2 — Specialized named subagents (pi-subagents pattern)

Second piece of work: turn the generic fan-out tool into a specialized named-agent system like pi-subagents — named agents (scout, reviewer, oracle, worker, …) each with their own model, thinking level, tool allowlist, system prompt, resource scope, and fan-out permission. New module packages/runtime/src/subagent-profiles.ts (the resolver + builtin bodies); the rest is wiring in subagent-extension.ts.

Decision Builtin agents live in code, not on disk

general-purpose, scout, reviewer, oracle, and worker are defined as constants in subagent-profiles.ts — no filesystem read, always available, and testable. User/project agents are still loaded from .md files and can override a builtin of the same name. (pi-subagents ships its builtins as bundled files; in-code is simpler for an embedded runtime and avoids a packaging step.)

Decision Resolution order & agent directories

Precedence is builtin < user < project < per-call overrides (later wins). User agents: <agentDir>/agents/*.md (the pi-agent dir, honk's equivalent of ~/.pi/agent/agents). Project agents: the nearest .honk/agents directory at or above the working dir. Assumption I chose .honk/agents for the project scope (the spec didn't name it); pi-subagents uses .pi/agents. Easy to change or additionally support if you'd prefer .pi/agents. Per-call agentScope (user/project/both) gates which scopes are read; default user.

Decision System prompt is appended in-process, not replaced

The load-bearing mechanism: DefaultResourceLoader({ appendSystemPrompt: [...] }) — the in-process SDK equivalent of pi-subagents' --append-system-prompt CLI flag (confirmed in the 0.78.1 dist, resource-loader.d.ts). I append (the profile body + a short boundary note that the child is a subagent and shouldn't try to converse with the user) rather than replace, so the base pi prompt — tool docs, safety guidance — survives. Tradeoff a profile can't fully replace the base prompt; if that's ever needed, the loader also exposes systemPrompt (replace) and override callbacks. The prompt is fixed at child-session creation (no mid-run rewrite); dynamic context goes through the task message via the new context param.

Tradeoff Tool allowlists drop read; specialists use bash

honk force-excludes the pi builtin read (DEFAULT_EXCLUDED_TOOL_NAMES in thread-agent-runtime) — it reads via its fff extension, which children don't load. So the builtin profiles use grep/find/ls/bash for read-only work (bash reaches file content) and add edit/write for the worker. Including read in an allowlist would be a silent no-op. This matches pi-subagents' own reviewer/scout (which include bash). Consequence: a "read-only" reviewer technically has bash and could modify — we rely on the system prompt to keep it read-only, as the reference does.

Decision The tools option drives the allowlist; policy field left empty

ThreadAgentRuntime.create derives the session's effective allowlist from its tools option (and re-derives policy.allowedToolNames from it internally), so I pass tools: profile.tools and leave policy.allowedToolNames: [] — both because it's overridden anyway and to avoid fighting the branded TrimmedNonEmptyString[] policy type. A null profile tools means "inherit the parent's default tools".

Decision Depth guard is in-process (no env/subprocess to carry it)

pi-subagents propagates depth via an env var to spawned child processes. honk runs children in-process, so depth is threaded explicitly: createSubagentExtension({ depth, maxSubagentDepth }). The factory refuses to register the subagent tool when depth >= maxSubagentDepth, and a child is handed its own fan-out extension only when its resolved tools include "subagent" and it's still below the cap. A profile's maxSubagentDepth can only tighten the global cap (default 2). No builtin includes subagent in its tools, so by default subagents cannot themselves fan out — exactly the requested behavior.

Decision Model resolution falls back to inheritance

A profile/override model string ("provider/id" or a bare id) is resolved against the registry (ModelRegistry.find/getAll); if it's absent or unknown, the child inherits the parent's current model rather than erroring. Same fall-through for thinking level and tools.

Deferred to follow-up PRs

Deferred · review #4 Move child execution off the Electron main thread

Today every subagent runs as an in-process ThreadAgentRuntime.create() on the main thread (up to 4 + the parent), so heavy parallel work still saturates the one event loop even after the IPC fixes. Two routes: (a) host DesktopRuntimeHost in an Electron utilityProcess and forward host events over its MessagePort; (b) genuinely spawn pi --mode json -p --session <fork> per child — pi's own sample design and pi-subagents' design. Risk lifecycle/teardown, sharing auth + model registry across processes, transcript transport. Starting point: the host is instantiated at packages/desktop/src/ipc/methods/runtime.ts:41; the child path is subagent-extension.ts → ThreadAgentRuntime.create.

Deferred · review #6 Unify the subagent UI on the chat renderer + one click

The subagent transcript renders through a bespoke flat tray instead of the main timeline's StepRenderer/markdown tree, and reaching a running child takes two clicks (expand the work-group collapsible, then click the row). Cursor's lesson: a subagent is a composer — same store, same renderer — reachable one click from a persistent pill. Starting point: packages/app/src/components/chat/composer/subagents/subagent-tray.tsx already converts some transcript items to TimelineStep and calls StepRenderer; extend that to the whole transcript and replace the collapsible→row→tray path with a single persistent affordance.

Deferred · review #1 Forward deltas, not accumulated message text

The base (non-subagent) assistant stream still emits one runtime-event per chunk whose data holds the full accumulated message — inherent O(n²) on that channel for a single long message (now halved by dropping raw, and its render path is covered by the coalesced display-timeline, but the channel cost remains). The real fix is to consume assistantMessageEvent.delta (pi's documented consumer pattern) and have the host's incremental message projection accumulate from deltas. Deferred because event.text being full-accumulated is load-bearing across the projection and many session-logic.ts consumers read payload.data — a semantic refactor, not a surgical one.
