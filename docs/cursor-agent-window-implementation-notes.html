00TL;DR & the StyleX question

Cursor is not on StyleX.

The premise "they recently migrated to StyleX" is refuted with hard evidence: zero stylex / @stylexjs / stylex.props literals in the 61 MB bundle, and zero StyleX atomic classes (.x1abc2d) in the CSS — the x800000000 hits seen on first probe were hex-color false positives. Cursor's workbench is a three-layer plain-CSS system: VS Code base (--vscode-* custom props) → a bespoke semantic --cursor-* token layer (radius/spacing injected at runtime via 6,621 setProperty calls) → a Tailwind v4 utility layer (@layer theme/base/components/utilities, --tw-* vars). Class names are human-readable and stable (.ui-tab-system-tab, .editor-panel-tab-header-row) — no CSS-Modules hashing.

So: how hard would Tailwind → StyleX be for Honk, and should we?

Recommendation: don't. It is a large, multi-week mechanical migration with high regression risk and little payoff — and crucially, the product we're chasing (Cursor) isn't even on StyleX.

Factor Reality for Honk
Surface area ~109 files in packages/app/src use className=, almost all Tailwind utility strings. Every one becomes stylex.create + stylex.props.
Toolchain Honk is Vite 8, no Babel. StyleX's first-class compiler is a Babel plugin; on Vite you rely on community unplugin-stylex/rollup shims of varying maturity — risky on a bleeding-edge Vite 8.
Overlap of benefits Tailwind v4 already gives atomic CSS, build-time dedup, and typed design tokens via @theme + CSS custom properties — the same wins StyleX sells. Marginal gain.
Dynamic styles Honk leans on data-* attribute variants + class concat (Cursor does the same). StyleX would force a different conditional-style idiom across the whole codebase.
What Cursor actually does Plain CSS classes + --cursor-* semantic tokens (resolving to theme vars) + runtime-injected scale. This is what Honk already has the shape of with Tailwind v4 + CSS vars.
Better move than a migration: formalize Honk's token layering to mirror Cursor — a base palette → a --honk-* semantic layer (every component references these, never raw colors) → a --glass-* surface layer with vibrancy on/off variants, all exposed to Tailwind via @theme. See §04 for the exact values. If you want StyleX's co-located typed styles, adopt it incrementally for new complex components only — never a big-bang rewrite.

Three load-bearing findings about the shell

  1. The top bar never re-renders on maximize because it's a position:fixed overlay; the only reaction to the maximize flag is a requestAnimationFrame that re-parents a stable DOM node (it never tears down the title). See §01.
  2. Fullscreen doesn't push the left sidebar because maximize hides only the editor leaf of the grid and grows the aux-bar by exactly width − sidebar − activitybar − unifiedSidebar. The sidebar's grid cell is never hidden or resized — you must toggle it off separately. See §01.
  3. Honk's 3-shell split is already a faithful port. RightAside = GlassRoot, RightAsideFrame = EditorPanelResizeShell, RightWorkbenchContent = EditorPanelMainShell. The remaining gaps are a generic tab-renderer registry and consolidating over-subscription. See §06.

01Maximize / fullscreen mechanism — the core question

Why the top bar only re-positions (CSS), never re-renders

Two layers combine:

(a) The titlebar is a fixed overlay — nothing to relayout

In Cursor's unified ("no-titlebar") layout the titlebar part is pulled out of flow and pinned over the grid. Maximizing the chat changes its box not at all: 

body.no-titlebar-layout .monaco-workbench .part.titlebar{
  position:fixed !important; top:0 !important; left:0 !important; right:0 !important;
  height:34px !important; z-index:10000 !important;
  background:transparent !important; pointer-events:none !important;
}
/* grid runs full-height beneath the fixed bar; parts clear it with padding-top:35px */
body.no-titlebar-layout .monaco-workbench > .monaco-grid-view{ height:100% !important; top:0 !important; }
body.no-titlebar-layout .part.auxiliarybar > .content .split-view-view{ padding-top:35px; }

The bar's three regions are pointer-events:auto islands floating over a full-width -webkit-app-region:drag strip. The only agent-mode rule is a no-op transform reset on mac.

(b) The maximize flag triggers a re-parent in a RAF, not a re-render

The titlebar registers exactly one subscription keyed on the agentChatMaximized / glassEditorPanelFullscreen context key. It schedules a single coalesced RAF that calls doUpdateActionsPositioning(), which moves an existing toolbar node between containers and toggles marker classes — it never creates or destroys widgets: 

// titlebar ctor — the ONLY reaction to the maximize key  @60057154
const o = new Set([j4e.key]);
this._register(this.contextKeyService.onDidChangeContext(a => {
  a.affectsSome(o) && requestAnimationFrame(() => this.updateActionsPositioning());
}));

// doUpdateActionsPositioning — re-parent + class toggle, no widget churn  @60079192
const mode = this.getLayoutActionsPositioningMode();        // "titlebar" | "editor-tabs"
target.insertBefore(this.actionToolBarElement, this.windowControlsContainer); // RE-PARENT a live node
this.actionToolBarElement.classList.toggle("in-titlebar",  mode==="titlebar");
this.actionToolBarElement.classList.toggle("in-editor-tabs", mode==="editor-tabs");
Why re-parent, not re-render? The toolbar is a live monaco-toolbar widget bound to menu state. Re-rendering would drop hover/focus, re-resolve menus, and flicker. Re-parenting a stable node preserves widget identity and lets CSS handle the visual move. The maximize/minimize icon swap is purely declarative (toggled: equals(agentChatMaximized,true) → icon: maximize/minimize) — no remount. The chat body never receives the boolean as a prop.

Why fullscreen does NOT push the left sidebar away

Mechanism: maximize hides only the editor leaf of the grid's middle horizontal branch, then grows the aux-bar to consume exactly the editor's freed width — computed by subtracting the still-present sidebar/activitybar/unified-sidebar widths. The left sidebar's grid cell is never hidden and never resized, so it stays put until the user toggles it off separately.

isUnifiedMode() && (e ? mainContainer.classList.add("agentmode") : mainContainer.classList.remove("agentmode"));

setViewVisible(view,false) in the SplitView grid sets the view's size to 0 but keeps the view item (it caches a restore size); the grid topology is untouched, so the sidebar leaf keeps its slot and width. On maximize, Cursor snapshots pre-maximize sizes as plain fields and only hides the sidebar when skipHideSidebar is false: 

// setUnifiedMaximizeState(true) — snapshot, then hide editor+panel (sidebar only if not skipped)  @43276669
this.sidebarWidthPercentageBeforeMaximize      = sidebarSize.width / mainW;   // restore later
this.auxiliaryBarWidthPercentageBeforeMaximize = auxSize.width / mainW;
this.unifiedSidebarWidthPercentageBeforeMaximize = uSize.width / mainW;

this.setEditorHidden(true, true);
// tail: wasMaximized = e; agentChatMaximizedContext.set(e); stateModel.save();
agentmode vs nomaineditorarea: both are chrome hooks, not layout drivers (sizing is the grid math above). nomaineditorarea (editor hidden, any mode) removes inner borders that framed the now-absent editor. agentmode (unified mode + editor hidden, i.e. true chat-maximize) re-skins the aux-bar title into a draggable agent titlebar and resets the mac action-cluster transform. Neither sets width/position on the sidebar.

Full state flow on toggle

  1. Invoke workbench.action.maximizeChatSize → reads current key, fires telemetry, calls layoutService.toggleUnifiedMaximizeState().
  2. Decide target = !(maximized ?? wasMaximized ?? !editorVisible); await setUnifiedMaximizeState(target).
  3. Mutate grid: snapshot *BeforeMaximize fields → hide editor (+panel, +maybe sidebar) via per-leaf setViewVisible → regrow aux bar.
  4. Write the context key once (_recomputeAndEmit syncs 4 keys + fires onDidChange).
  5. Readers (all imperative): titlebar RAF re-parents actions; aux-bar title part restyles; the toggle icon swaps declaratively; the body just gets wider because its grid cell grew.

Every shell component is React-Compiler auto-memoized (const t=Tt(N) cache arrays + t[i]!==dep guards); some are also React.memo-wrapped (noted below). The single subscription primitive is uxe(service, selector) = useSyncExternalStore with a structural-equality (V9t) bail-out — one layout service, many consumers each subscribing to a disjoint slice.

03State model & subscribe-at-consumer

Three DI services own all shell state; each is a hand-rolled emitter store (not observables):

Service Owns Subscription surface
Tab manager (VNf) per-group tabs / stableTabs / activeTabId subscribe(type, cb) (type-filtered), subscribeAll(cb)
Tab persistence (glassTabPersistenceService) high-level ops: createTerminal/Browser/Canvas/File, activateDiffTab… consumers subscribe to .manager.subscribeAll
Layout service (glassLayoutService) one immutable snapshot + pre-maximize fields onDidChange(snapshot ⇒ …)
Command bridge (glassCommandBridgeService) typed pub/sub bus (88 event types) + late-listener replay buffer onEmitted(type, cb) via the Ap() hook

The anti-cascade mechanism (two layers)

(a) Selector hook with structural dedupe

function uxe(service, selector){               // = useSyncExternalStore
  const subscribe   = cb => service.onDidChange(cb);
  const getSnapshot = () => {
    const next = selector(service.getSnapshot());
    return V9t(prev, next) ? prev : (prev = next, next);  // structural-equality bail-out
  };
  return ac(subscribe, getSnapshot);
}

A component selecting {editorPanelVisible, editorPanelFullscreen} re-renders only when those change. Sash drags fire onDidChange but the selector returns a V9t-equal object → React bails out. Width never reaches React — it's applied via CSS variables / direct style writes.

(b) Inline event-type + id filtering inside the subscribe callback

// TabSystemTabs subscribes to everything but filters before notifying React  @4357518
subscribe = notify => manager.subscribeAll(ev => {
  TAB_LIST_EVENTS.has(ev.type) && "groupId" in ev.payload && ev.payload.groupId === myGroup && notify();
});
// per-tab consumers additionally filter ev.payload.tabId === myTab
// active tab is its OWN slice: manager.subscribe("activeTabIdChanged", notify)
Net effect: a tabDirtyChanged on tab X in group A re-renders neither group B's bar nor tab Y's item. The toolbar subscribes only to activeTabIdChanged (or a V9t-deduped layout slice). The composition root subscribes to nothing volatile. This is the single biggest reason Cursor's toolbar/topbar don't churn.

Context keys + event bridge (instead of prop-threading)

One state mutation (_recomputeAndEmit) atomically (1) rebuilds the immutable snapshot, (2) syncs context keys (glassEditorPanelVisible, glassEditorPanelFullscreen, …) for keybinding when/toggled clauses, (3) fires onDidChange for React consumers. Cross-tree UI intent (e.g. opening the + menu) flows through the command bridge: the + button emits editorPanelNewTabMenuRequested; the header Ap()s it into local useState — emitter and listener never share a render path.

The design system is not a shipped @anysphere/ui npm package.

The + NewTabMenu (_SI)

A global menu for spawning dynamic tab kinds: search input placeholder "Open any file, URL, ..."; empty-query items = Changes / Terminal / Browser / File / Canvas (each → a glassTabPersistenceService create method); with-query = URL → browser tab, fuzzy Files → open file, web suggestions.
