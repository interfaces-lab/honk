Research date: May 18, 2026

Should Multi use Solid.js to improve rendering?

Short answer: not as a broad migration. Cursor is useful evidence, but it points to selective renderer boundaries, not replacing a React app wholesale. Multi should keep the React/TanStack Router shell and use Solid only as an isolated island if profiling proves a specific hot path needs finer-grained DOM updates.

Recommendation

Stay on React 19 + React Compiler + TanStack Router. Do not migrate the app to Solid for a general rendering win.

Best Solid use case

A bounded high-frequency surface such as live token/status rows, terminal-adjacent controls, or a future canvas/workbench widget.

Main risk

Mixed framework ownership adds state, context, styling, event, test, and build boundaries. Without a measured hotspot, that cost is not justified.

Solid mount shape

The Solid workbench adapter creates a host div, injects service context, renders a Solid component, catches render errors, and returns dispose() / focus().

React bridge shape

The React bridge exposes a Solid component wrapper around a React component, creates or registers a React root, updates it when the component, props, or portal root changes, and unmounts on cleanup.

Cursor's architecture is evidence that mixed rendering can work, but only with explicit ownership boundaries. The bridge is extra infrastructure, not free performance.

Multi App Facts

Multi is currently a React renderer using TanStack Router and TanStack Query. The dependency and entrypoint evidence is direct:

  • packages/app/package.json has @tanstack/react-router, @tanstack/react-query, @tanstack/react-virtual, react, and react-dom as app dependencies.
  • packages/app/src/main.tsx imports React, react-dom/client, and RouterProvider, then mounts the app with ReactDOM.createRoot(...).render(...).
  • packages/app/src/router.ts constructs the router with createRouter from @tanstack/react-router and wraps route rendering in QueryClientProvider.
  • packages/app/vite.config.ts already wires tanstackRouter(), React, and reactCompilerPreset().
  • solid-js is present in pnpm-lock.yaml only through TanStack devtools dependencies; the app does not depend on @tanstack/solid-router or @tanstack/solid-query.

Current rendering posture

Area Observed implementation Impact on Solid decision
Chat route shell ChatView has many granular store selectors, local refs, and memoized derivations. This already follows the same principle Solid would provide: narrow subscription boundaries.
Timeline MessagesTimeline is memo-wrapped, derives stable rows, uses @tanstack/react-virtual, and owns scroll state. The biggest list-rendering win is already virtualization, not a framework swap.
Router state Route files are intentionally thin and route views bridge client-only state. Replacing React Router bindings with Solid Router bindings would touch a lot of shell code for little direct rendering benefit.
React effects Existing project docs require derived state to stay in render and direct useEffect to be avoided in new React code. The app is already moving toward reactive discipline inside React.

External Facts

  • TanStack Router has current React and Solid documentation, including Solid-specific guides and examples. TanStack describes Router as type-safe routing for React and Solid applications.
  • Solid's official docs describe fine-grained reactivity as targeted updates to the changed UI data rather than re-executing a whole component.
  • TanStack Router's render optimization docs already recommend fine-grained selectors and structural sharing for route hooks that return new object values.
  • React Compiler is already in this app's Vite config. React's docs say the compiler adds automatic memoization for update performance and can skip cascading re-renders when analysis allows it.

Options

Option What it means Upside Downside Verdict
Keep React and optimize Stay on React/TanStack Router, profile hot paths, tighten store selectors, use router structural sharing where safe, keep React Compiler healthy. Lowest risk. Fits current code and project rules. Does not give Solid's runtime model. Default path.
Solid island Add one isolated Solid-rendered surface behind an explicit mount adapter, similar to Cursor's workbench boundary. Lets us test Solid where high-frequency updates are measurable. Requires bridge code for services, stores, styling, events, cleanup, and tests. Only after profiling.
Full Solid migration Replace React app bindings with Solid Router/Solid Query and port UI components. Maximum framework consistency if the app were greenfield. Large rewrite, package replacement, test rewrite, component API churn, and uncertain payoff. Do not do this for rendering alone.

Recommendation

Keep React as the product renderer. Treat Solid as a possible performance tool for a measured, isolated surface, not as a replacement for the existing shell.

The reason is concrete: Multi's hot UI is not a simple unvirtualized list that React is naively rerendering. The chat timeline is already virtualized and memoized. The route layer is already thin. The build already includes React Compiler. Cursor's own bundle shows that mixed rendering works by adding a Solid mount adapter, a service-context layer, reactive-storage wrappers, and a React bridge. That is useful only when the target surface is worth owning as a separate renderer.

What I would do next before any implementation

  1. Capture React Profiler traces for the exact slow workflow: streaming assistant text, timeline scroll, command palette search, model picker, diff panel, or terminal drawer.
  2. For route/search churn, try TanStack Router structural sharing on the specific hook or router default only after confirming returned data is JSON-compatible.
  3. For chat/timeline churn, inspect whether message row identity, derived maps, markdown rendering, or store selector breadth is the actual bottleneck.
  4. If a hotspot remains high-frequency and isolated, prototype a Solid island with a Cursor-style mount boundary and a deliberately tiny API: props in, events out, explicit dispose.

Do not mix Solid and React inside the same ownership surface. The safe shape is a hard boundary: one framework owns a subtree, the other passes data and callbacks across a small adapter.
