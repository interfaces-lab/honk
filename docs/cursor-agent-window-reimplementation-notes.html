Cursor Agent-Window Shell — Canonical Reimplementation Notes

A single source of truth for reimplementing Cursor's agent-window shell (top nav, right-aside second nav, and the maximize/fullscreen mechanism) on top of Honk's current React shell. Everything below is reverse-engineered from workbench.desktop.main.js/.css by five engineers; byte offsets (@…) are into those files. Where a report flagged something unfound, it is marked TODO.

1Goal & summary

Reach visual + behavioral parity with Cursor's agent window for two hard requirements, mapped onto Honk's existing AppShell tree:

  1. Zero top-nav re-render on fullscreen. Toggling maximize must not re-render the top nav or any header chrome — in Cursor the bar only re-positions via DOM re-parenting + class toggle, never re-renders, because the toolbar lives outside React (a VS Code WorkbenchToolBar).
  2. Stable, reflow-based layout. Entering fullscreen must collapse the left sidebar and the center/chat region out of layout so the right workbench reflows to fill the window — a genuine push-away, not an absolute overlay over a still-present sidebar.
The five canonical findings
  • Two top bars, mutually exclusive by mode (gbI ContentPaneTopBar for plain/empty chat; O7e.Bar tab-system bar for the tabbed agent window). Both 35px, share the same left-cluster primitives.
  • The second nav is per-panel, full-width above the split — one shared 36px header wrapper (KDp), one shared resize handle (Ojt) + hook (mzn), side-docked via data attributes.
  • Maximize = hide three siblings together, grow one. It is a grid reflow (setViewVisible(view,false)), NOT a CSS overlay. agentmode/nomaineditorarea classes are cosmetic only.
  • Toolbar "no re-render" = move one live DOM node between two host containers (appendChild/insertBefore) + class toggle, coalesced to one RAF. The React-faithful analogue is a portal with a swappable target.
  • One state owner, many slice subscribers. Almost all glass state is in a single hook (hPI) holding ~3 useState; everything else is read from services via memoized, stable-identity hooks (kn, uxe) at the consumer.

Honk already mirrors much of #1 and #5 (memoized RightWorkbenchHeader, the RightAside → RightAsideFrame → RightWorkbenchContent split, subscribe-at-consumer fullscreen leaves, the imperative data-shell-fullscreen-target write). The outstanding gap is requirement #2: Honk's fullscreen is an absolute overlay, and the left sidebar does not collapse. See §8 bug + fix.

5Fullscreen / maximize mechanism

There are two independent state machines: (A) toolbar host — where the action cluster lives (titlebar vs in-tab), the zero-re-render reposition; and (B) geometry — which panes exist and their sizes, the reflow. Do not conflate them.

5.1 PART A — the zero-re-render reposition (updateActionsPositioning)

Wired to context keys agentChatMaximized (j4e) and panelMaximized (n2i). On change it schedules one requestAnimationFrame, debounced via pendingActionsPositioningUpdate.

The coalescing RAF gate + the actual DOM mutation (@60079192) 
// registration @60057800 — ao(this.element) resolves the correct Window (aux window aware)
this._register(this.contextKeyService.onDidChangeContext(a=>{
  a.affectsSome(new Set([j4e.key]))&&ao(this.element).requestAnimationFrame(()=>{this.updateActionsPositioning()})

updateActionsPositioning(){                              // pure coalescing gate
  this.pendingActionsPositioningUpdate!==void 0 && ao(this.element).cancelAnimationFrame(this.pendingActionsPositioningUpdate),
  this.pendingActionsPositioningUpdate=ao(this.element).requestAnimationFrame(()=>{
    this.pendingActionsPositioningUpdate=void 0, this.doUpdateActionsPositioning() })
}

doUpdateActionsPositioning(){
  if(this.actionToolBarElement){
    const n=this.getLayoutActionsPositioningMode(),       // "titlebar" | "editor-tabs"
          e=this.findTopRightMostGroupIncludingAuxBar(),

}
Why the React toolbar subtree does NOT re-render
  • The action toolbar is not a React subtree — it is a VS Code WorkbenchToolBar (a real DOM node owned by the imperative titlebar Part).
  • The node is moved, not recreated: appendChild/insertBefore re-parent the existing element. DOM identity, rendered children, and handlers survive the move — no reconciliation, no setState, no new props.
  • Only class toggles (in-titlebar/in-editor-tabs) + an occasional display flip are applied; cosmetic CSS only.
  • The RAF debounce coalesces the burst of context-key/visibility events into a single mutation per frame.
React-faithful analogue: a portal whose target container is swapped. Render the toolbar once, mount via createPortal(toolbar, targetContainer); swap the target (titlebar ref ↔ in-tab ref) from a memoized selector over {editorVisible, chatMaximized, rightSidebarHidden, rightEdgeEmpty, …} (exactly like $qg → XLx/SGS/ZLx). Debounce the swap to one RAF.

The mechanism is GRID REMOVAL + REFLOW — never an overlay. The workbench is one SerializableGrid; all parts are sibling leaf nodes in one horizontal split row ({type:"leaf",data:{type:"workbench.parts.sidebar"},visible:…}). Hiding a part calls setViewVisible(view,false) → distributeEmptySpace(); layoutViews(); saveProportions() — a real reflow that reclaims the hidden leaf's space for its siblings. Maximize-enter explicitly calls setSideBarHidden(true), so the left sidebar is removed and its width reclaimed.

Why a clone shows fullscreen NOT pushing the left sidebar until toggling off: the clone hid the editor (added agentmode/nomaineditorarea, grew the aux pane) without also issuing the sibling setSideBarHidden(true) grid removal — OR it overlaid the maximized chat on top instead of removing the sidebar leaf. agentmode/nomaineditorarea are cosmetic only (border/padding removal); they set no width/position/transform on any pane.

Invariants worth copying: never let all of {editor, panel, chat} hide (ensureMainContentAreaVisibleInvariant force-exits maximize); remember manual sidebar toggles during maximize (sidebarVisibleBeforeMaximize); auto-exit maximize if the user re-opens a conflicting pane (setPanelHidden early-return). Memo / context boundaries & imperative escapes

  • Split contexts by cadence. nLA → DIk.Provider carries only the composer's pendingPrompt/mode/slashCommand slice, memoized on those fields — typing re-renders only composer consumers, never the agent list or panels.
  • Toggles flip DOM/grid imperatively; React catches up only via subscription. Resize writes style.width on every mousemove (mzn), committing to the store only throttled — zero React renders during a drag.
  • Event bridge for imperative→declarative. glassCommandBridgeService (Jd, impl YOg) with emit/onEmittedInternal, deferred events (drained on mount), and envelope routing ({requestId, targetWorkspaceId, targetAgentId}). React side Ap() stores the handler in a ref so it never re-subscribes when the callback identity changes. E.g. editorPanelNewTabMenuRequested flips one local useState → only that component re-renders.
  • Refs for transient, state for committed. useRef for values that survive renders but must not trigger them; useState reserved for values that should re-render exactly one subtree.

8.2 The left-sidebar-not-collapsing bug — root cause & exact fix

Bug Entering fullscreen does not push the left sidebar away — it stays at its width until the user toggles fullscreen off. The workbench fills the window by sitting on top of the still-present sidebar.

Root cause — current rules in shell.css @336–355 make the workbench an absolute overlay, not a reflow:

Current (overlay) CSS — shell.css @336–355 
.agent-window[data-shell-fullscreen-target="right-workbench"] .agent-window__workbench {
  position: absolute;          /* ← OVERLAY: leaves the flex flow */
  inset: 0;
  z-index: var(--z-index-shell-overlay-panel);
  width: 100%;
  min-width: 0;
  max-width: 100cqw;
  border-left-color: transparent;
}
.agent-window[data-shell-fullscreen-target="right-workbench"] .agent-window__sidebar {
  width: 0;                    /* ← loses to the inner w-[min(...)] div + shrink-0 */
  min-width: 0;
  border-right-color: transparent;
}
.agent-window[data-shell-fullscreen-target="right-workbench"] .agent-window__workbench-body {
  width: 100%;
}

Two failure modes, both confirmed against app.tsx:

  • (a) inset:0 resolves against the wrong ancestor. .agent-window__workbench (the RightAsideFrame <aside>) lives inside the intermediate <div className="relative flex … flex-row"> wrapper at app.tsx line 907, not .agent-window. The overlay covers only that inner row — and because it's position:absolute, it no longer participates in flex, so the sidebar (a flex sibling on .agent-window itself, line 902) is never reflowed.
  • (b) .agent-window__sidebar{width:0} loses. The <aside> is shrink-0 and its width comes from the inner body div w-[min(var(--honk-shell-left-width),100cqw)] (line 223). Setting the aside's width:0 does not collapse the inner fixed-width div, so the column keeps its width.
Fix — replace overlay with a genuine reflow (Cursor §5.2 faithful) Keep the workbench in flex flow; collapse both siblings out of flow via the root attribute. No position:absolute, no z-index. 
/* shell.css — replace the @336–355 block */
.agent-window[data-shell-fullscreen-target="right-workbench"] .agent-window__workbench {
  flex: 1 1 auto;            /* grow into the freed space — NO position/inset/z-index */
  min-width: 0;
  max-width: 100cqw;
  border-left-color: transparent;
}
/* collapse the LEFT sidebar column out of flow (target both the aside AND its inner width div) */
.agent-window[data-shell-fullscreen-target="right-workbench"] .agent-window__sidebar {
  flex-basis: 0; width: 0; min-width: 0; overflow: hidden; border-right-color: transparent;
}
.agent-window[data-shell-fullscreen-target="right-workbench"] .agent-window__sidebar > div {
  width: 0;                 /* the inner w-[min(...)] body must also go to 0 */
}
/* collapse the CENTER region out of flow (needs a stable class hook — see step 2) */
.agent-window[data-shell-fullscreen-target="right-workbench"] .agent-window__center {
  flex-basis: 0; width: 0; min-width: 0; overflow: hidden;
}
.agent-window[data-shell-fullscreen-target="right-workbench"] .agent-window__workbench-body {
  width: 100%;
}
Transition width/flex-basis on the siblings (they already have width transitions) so the workbench expands as they collapse — a clean reflow, no overlay, no shift. This mirrors Cursor calling setSideBarHidden(true) + setEditorHidden(true) together against one grid row.
