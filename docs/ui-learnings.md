# UX learnings from the current interface (round 2)

What the current UI teaches, per surface — collected by the round-2 opus sweep as design input for
the wireframe boards and, later, the `.design/*` references (ADR 0023 §4). These are lessons, not
rules: the boards argue with them, and the winning board becomes project-local design guidance.


## composer

- **The composer is a 3,155-line god-component held together by refs and manual sync** — Treat the composer in the rewrite as a thin view over projected state. Do not reimplement the editor<->prompt reconciliation (syncEditorToPrompt, forceSyncGeneration, expand/collapse cursor math, and the half-dozen null-render "...Sync" components keyed by concatenated strings). Most of this bulk exists to paper over a rich-document editor fighting a plain-text model.
  Evidence: packages/app/src/components/chat/composer/input.tsx (3,155 lines); prompt-triggers.ts implements expandCollapsedComposerCursor/collapseExpandedComposerCursor to map chip positions between text and document; input.tsx has ComposerMenuHighlightSync/ComposerPendingInputPromptSync/ComposerDraftResetSync/ComposerMenuAnchorObserverSync etc.
- **Dual source of truth: plain text vs. persisted rich-text JSON** — ADR 0013 wants raw text + Prompt Tokens on the wire. The current composer keeps livePrompt (string) AND serializes the whole Lexical document to richTextJson (debounced) into the draft store, forcing constant cursor translation. Model tokens as first-class inline atoms over a plain-text buffer and persist text + token spans, not a document.
  Evidence: input.tsx scheduleRichTextDraftPersistence/persistRichTextDraftNow write richTextJson; chat-drafts.ts ComposerThreadDraftState has both prompt and richTextJson; serialization.ts/prompt-triggers.ts exist solely to reconcile the two.
- **Large parts of the composer are wired to an all-rejecting stub and can never fire** — Model/agent-mode picker, queued-items panel (list/edit/reorder/send-now), extension-UI panel, /compact, and past-thread mentions all route through honk-runtime-api / agent-runtime-store, which receive no events after the core cutover. They look alive but are dead. The rewrite must rebuild the queue and the creation-time model/thinking selection on the core wire and delete the desktop-runtime coupling.
  Evidence: lib/honk-runtime-api.ts (rejectUnavailable + empty snapshot + onHostEvent no-op, isDesktopRuntimeApiAvailable()===false); agent-runtime-store.ts never calls setSnapshot/applyHostEvent; use-thread-composer-queue.ts reads snapshot.queuedFollowUps (always []); runtime-turn-dispatch.ts compactRuntimeThread throws; GAPS.md queue/session-file gaps.
- **Approval UI is threaded through the composer's whole state machine — and ADR 0007 deletes it** — Removing approvals is a big simplification, not just a panel deletion: isComposerApprovalState gates the placeholder, disabled input, attachment button, footer action row, and header. Ask-question and plan proposals are the survivors and should render as timeline Parts, not composer headers.
  Evidence: input.tsx isComposerApprovalState branches ~10 render decisions; pending/approval-panel.tsx + approval-actions.tsx; chat-view.tsx dispatches thread.approval.respond which GAPS lists as unsupported; ADR 0007.
- **Cursor-parity is the composer's explicit design north star** — The menu ranking, collapsed "Show N more" sections (3 skills / 3 files / 5 threads), single 300px menu shell, mode-prefix pinning, and glass surfaces are deliberate Cursor mimicry. Preserve the interaction model, but rebuild the anchor/positioning — the virtual-anchor + MutationObserver + anchorRevision hack is fragile.
  Evidence: command-menu/menu.tsx comments ("Cursor parity: collapsed sections show 3 items above a Show N more row", "Cursor parity: both menus use one 300px shell"); input.tsx ComposerMenuAnchorObserverSync retries on animation frames and bumps a revision to reposition Floating UI.
- **Model + thinking are already creation-time-only in-thread, matching ADR 0014** — Keep the split the composer already has: a rich picker in the new-agent/draft composer and a read-only chip in an existing thread. There is no cycleModel to remove. The only work is re-wiring the creation picker to write model+thinking into the thread-create command instead of the dead preferences store.
  Evidence: input.tsx renders ComposerAgentModePicker only when isNewAgentComposer, else ComposerReadOnlyAgentModeChip; GAPS.md "core pins model and thinking at thread creation".
- **Delivery modes are hidden behind a settings-driven button label** — ADR 0005 makes queue/steer/interrupt first-class, but today the only surfaced control is agentWindowSendWhileStreamingBehavior changing the send button's label while running, plus an optimistic (and dead) queue list. Make delivery mode a legible per-send affordance and project queued input as real state.
  Evidence: input.tsx PrimaryActionControls picks label from sendWhileStreamingBehavior (queue/stop-and-send/send-immediately); queue panel sources from the never-populated agent-runtime-store.
- **Desktop-only couplings quietly break remote parity** — Skills, past-thread mentions, path preview, worktree prep, and branch name all gate on isDesktopRuntimeApiAvailable()/aux. ADR 0022 requires a phone/browser client to get every feature. Every composer capability must resolve through one remote-capable plane.
  Evidence: command-menu/menu.tsx gates skills and threads on isDesktopRuntimeApiAvailable(); input.tsx prefetch and selectComposerThreadMention early-return when the desktop API is absent; status bar branch name comes from desktop git (aux).

## conversation-timeline

- **The timeline reconciles two sources; one is now dead — collapse to a single source** — timeline-render-items.ts (2176 lines) plus stabilize/regroup passes exist almost entirely to merge committed core WorkLogEntry rows with a parallel legacy runtime display timeline that 'flips source mid-turn as persistence catches up.' Post-cutover the runtime store never fires, so all runtime-* rows are dead. The rewrite should build the timeline from ONE source (core Parts); doing so deletes the seam-survival logic, the runtime/work de-dup, keepTailGroupRunning source-flip handling, and the runtime-only special groups.
  Evidence: Comments 'a turn's steps flip source independently as persistence catches up, so the run must survive the seam'; runtime steps come from useAgentRuntimeStore (activeRuntimeDisplayTimeline/selectRuntimeEventsForThread), which the audit brief and GAPS.md confirm receives no events.
- **The grouping/density engine is Cursor-parity-shaped and oversized** — Work-group collapsing carries three densities, explore-only thresholds, single-shell exceptions, thinking/command/await-monitoring/browser special groups, tail-group-keep-running, and narration release/peel — much of it justified by 'Cursor parity' rather than a chosen model. The running preview alone adds ResizeObservers, output-strip height math, and ~8 magic PX constants that the virtual-size estimator must mirror. The rewrite should define its own deliberate grouping contract (what collapses, when, what a running group shows) instead of porting Cursor's rules, and derive row heights from that contract.
  Evidence: Repeated 'Cursor parity (A4b)', 'Cursor zIb parity', 'Cursor kRm compact' comments; WORK_GROUP_* PX constants in step-renderer.tsx duplicated in estimateTimelineRowSize in messages-timeline.tsx.
- **User-message meaning is recovered by regex on rendered text** — Three separate heuristics parse structured meaning back out of the prompt string after the fact: git actions (resolveGitAgentActionFromPrompt), terminal context (TERMINAL_CONTEXT_HEADER_PATTERN), and the active-goal bar (/^Goal:/). These are fragile and only fire for exact shapes. Under ADR 0013 (raw text + Prompt Tokens), these should be first-class token/attachment types on the message, rendered directly — never re-derived from display text.
  Evidence: user-message.tsx (git action + terminal-context inline label matching), extractGoalText in chat-view.tsx.
- **Rich-text rendering is a whole second document engine that ADR 0013 removes** — The transcript renders message.richText through an ~880-line renderer that understands BOTH TipTap and Lexical node trees — a parallel implementation to the composer editor, just to show mentions/commands/skills/formatting. ADR 0013 makes the wire raw text + Prompt Tokens, so the stored editor document goes away. The rewrite needs exactly one render path over the token model shared with the composer, not two editor-doc parsers.
  Evidence: rich-text-message.tsx handles doc.type==='doc' (TipTap) and doc.root (Lexical) with duplicate renderTiptap*/renderLexical* trees; hasRenderableRichText gates it.
- **Branching exists in the data model but has no product UI** — The thread is a full entry tree; edit/resend creates sibling branches; the timeline correctly projects to the active leaf path. But the only way to see the tree or switch branches is a devtools panel — in the product, editing a message forks a branch the user can never navigate back to. This is a real gap the rewrite must close (a branch indicator/navigator on forked turns), not just a port.
  Evidence: ThreadTreePanel imported only by dev/router-devtools; no setLeaf/branch-switch call in components/chat; deriveThreadBranchView filters to leaf path only.
- **Removing approval (ADR 0007) is a cross-cutting deletion, not one panel** — Approval isn't isolated to approval-panel.tsx: pending-approval kinds thread through the grouping engine (a pending tool breaks out of its run), force 'detailed' density on the affected card, correlate approvals to running tools by kind, and drive shell auto-expansion. Deleting the approval concept lets the rewrite simplify grouping, density resolution, and shell expansion at the same time — plan proposals and ask-question (also ADR 0007) stay as ordinary Parts.
  Evidence: groupedStepHasPendingApproval/matchesPendingApprovalKinds in timeline-render-items.ts; resolveEffectiveToolCallDensity returns 'detailed' for pending approvals; ShellToolCall expansion keyed on approval.status in tool-renderer.tsx.
- **Context usage has two divergent renderers over one snapshot** — The same ContextWindowSnapshot feeds a ring-only meter with one popover (context-window-meter.tsx) and a thread-status bar with a colored category-breakdown popover (context-usage-bar.tsx), each with its own token formatting and empty state. Consolidate to one context-usage component with one formatting/empty-state source of truth.
  Evidence: ContextWindowMeter + ContextWindowUsageDetails vs. ComposerContextUsageBar + ContextUsagePopover; both re-implement percentage/token formatting and the auto-compacts note.

## onboarding-auth

- **There is no onboarding — first run is a void, then a failure page** — The current product has no first-run experience: the desktop window is hidden until Core is reachable, and the only auth screen a user ever sees is an error page ('Local authentication failed') with a Reload button. Accounts default to empty and Anthropic requires an out-of-band `claude login`. The rewrite should design an actual first-run that (a) shows connecting/spawning progress and (b) nudges the user to connect their first provider account.
  Evidence: desktop-core-manager.ts keeps the window hidden until revealMainWindow; __root.tsx has no pendingComponent; -root-route.tsx AuthenticationRequiredView only offers Reload/Open-in-Browser; settings-panels.tsx shows 'No accounts connected' only inside Settings.
- **Two different 'logins' are conflated for the user but split in code** — Transport auth to the local Core (bearer/pairing — the gate) and provider/model credentials (Accounts: Codex/Cursor/Claude Code) are distinct systems, but a new user's mental model of 'sign in' maps to provider login, which the gate never touches. Keep them cleanly separated in the rewrite, and make provider connection the emotional 'get started' step, with transport auth as invisible plumbing wherever possible.
  Evidence: environments/primary/auth.ts handles only the Core bearer/pairing gate; provider logins live entirely in core-auth-react-query.ts + the Accounts section of settings-panels.tsx; the gate page has no provider affordance.
- **Remote parity (ADR 0022) is endpoints-only with zero UI** — Pairing issuance, session listing/revocation, exposure-mode toggle, and manual token entry all exist as core/v1 endpoints or desktop IPC but have no rendered surface; a web client that lands without a #token= dead-ends on a Reload-only page. Since ADR 0022 makes remote parity absolute, the rewrite must build the owner-side 'connect a device' (pairing link/QR), a device-management panel, an exposure opt-in, and a web token-entry path from day one — these are the biggest gaps on this surface.
  Evidence: createServerPairingCredential / listServerClientSessions / revokeServerClientSession in auth.ts have no non-test consumers; getServerExposureState/setServerExposureMode IPC (server-exposure.ts, preload) is never called from packages/app; submitServerAuthCredential is exported but unwired; browserBootstrapUrl is null in web mode.
- **Connection status is invisible end to end** — Spawn/discovery, initial connect failure, and WS/SSE reconnect are all silent — a hidden window, a swallowed .catch, and a re-subscribe with no indicator. For phone and flaky-network clients this is unacceptable; the rewrite needs first-class connecting / reconnecting / offline / failed states shared across the client family, driven by the transport layer.
  Evidence: startCoreEnvironmentConnectionService in service.ts does createPrimaryCoreEnvironmentConnection().then(...).catch(() => undefined); subscribeWithReconnect + onStatus 'reconnecting' only reset the bootstrap gate; no offline banner exists anywhere in packages/app/src.
- **Delegated Anthropic auth has correct posture but a dead-end UX** — The Claude Code row is read-only and, when unavailable, literally tells the user to run `claude login` in a terminal — right per ADR 0016 (honk never stores an Anthropic credential) but a hard stop inside the product. The rewrite should visually distinguish 'delegated, managed in Claude Code' from honk-owned logins (Codex/Cursor, which have real in-app flows) and guide/deep-link the delegation rather than printing a shell command.
  Evidence: settings-panels.tsx Claude Code SettingsRow description: 'Uses your Claude Code login. Run `claude login` to connect.' with no control; ADR 0016 confirms the delegated posture.
- **The Codex device-code flow is the reusable login template** — CoreAuthFlowPanel is the one well-built auth interaction on this surface: open-verification-URI, copyable user code, waiting/error states, retry, cancel, 2s polling, and graceful fallbacks (clipboard-unavailable toast, external-open falling back to window.open). Keep it as the pattern any future OAuth/device flow reuses.
  Evidence: settings-panels.tsx CoreAuthFlowPanel + coreAuthSnapshotQueryOptions refetchInterval (2s while flow !== null) in core-auth-react-query.ts.

## settings

- **Settings live on four disjoint persistence planes** — One row UI hides four backends: server settings.json (aux, via server.updateSettings), client settings (local-api persistence/localStorage), appearance (localStorage + CSS vars in appearance-store), and agent prefs (dead runtime stub). use-settings.ts splitPatch routes each key by membership in ServerSettings.fields. For ADR 0022 remote parity the rewrite should collapse these onto one core-backed settings plane, keeping only genuinely per-device knobs (theme/fonts/vibrancy) client-local by intent, not by accident.
  Evidence: packages/app/src/hooks/use-settings.ts splitPatch + ensureLocalApi().server.updateSettings vs persistence.setClientSettings; stores/appearance-store.ts (separate localStorage plane); lib/honk-runtime-api.ts stub.
- **The Agents 'Pi runtime' section is a live-looking dead surface** — Agent mode, thinking level, and interaction-mode controls read from agent-runtime-store (never fed events) and write through readHonkRuntimeApi().updatePreferences, which only mutates a module-level fallback object and returns it. The controls appear interactive but persist nothing and never round-trip. The rewrite must never ship a control whose write target is a stub; every settings control needs a real, observable backing store.
  Evidence: lib/honk-runtime-api.ts updatePreferences mutates fallbackPreferences; AgentRuntimeSettingsSections reads useAgentRuntimeStore(state=>state.snapshot); isDesktopRuntimeApiAvailable()===false.
- **Model/thinking selection is framed as a global default, but ADR 0014 pins it at creation** — The Agent mode selector is a rich global 'default model for new sessions' with a nested model+effort submenu and availability gating. Under ADR 0014 (model+thinking pin at thread creation, no cycleModel) this belongs on the thread-creation surface, not a persistent app-wide preference. Keep the availability-gating and submenu affordances but relocate the decision to where a thread is born.
  Evidence: settings-panels.tsx AgentModeSelector description 'Default model for new sessions.' + AGENT_MODE_MODEL_DETAILS; deriveAgentModeAvailability/unavailableAgentModeReason.
- **Skill discovery is gated behind a runtime that is always unavailable** — The Skills panel is permanently stuck on its 'available in the desktop runtime' unavailable state because isDesktopRuntimeApiAvailable() is hardcoded false and listSkills returns an empty array. A skills browser only makes sense in the new product if skill discovery is served by the Core so a phone/browser client can see it too (ADR 0022); otherwise cut it.
  Evidence: lib/runtime-skills.ts + lib/honk-runtime-api.ts (isDesktopRuntimeApiAvailable→false, listSkills→{skills:[]}); SkillsSettingsPanel skillsUnavailableReason.
- **A fully-built 'Restore defaults' action has no trigger** — useSettingsRestore implements per-setting dirty detection, a confirm dialog that lists exactly what resets, and a remount signal — but nothing renders a button to call restoreDefaults; only restoreSignal is consumed. This is dead-weight complexity (a provider + context + hook) for an unreachable capability. The rewrite should either expose a single visible reset entry or drop the machinery entirely; don't ship provider plumbing for actions users can't reach.
  Evidence: settings-panels.tsx useSettingsRestore (changedSettingLabels, restoreDefaults); routes/-settings-route.tsx consumes only restoreSignal; grep finds no other consumer.
- **Live preview + reveal-when-dirty reset is the pattern to keep** — The strongest UX here: sliders/inputs with immediate previews (tool-call density renders real cards/rows, code font renders a tokenized diff, hue shows a live swatch) and per-row reset buttons that only appear when a value is non-default. This makes changes legible and reversible without clutter. Carry these patterns forward as the default for any preview-able setting.
  Evidence: tool-call-density-control.tsx ToolCallDensityPreview; appearance-controls.tsx CodeFontFamilySettingsRow preview + SettingsSlider swatch; settings-layout.tsx SettingResetButton (rendered only when dirty).
- **Desktop-coupled rows will break on remote clients** — Several rows assume a local machine: 'Open keybindings file' and 'Open logs folder' shell out to an external editor, the dock App Icon and updater are Electron-only, and the archived-row context menu uses the native OS menu. Under ADR 0022 the rewrite must sort these into inherent desktop chrome (updater, dock icon — keep, gated) vs settings that need a real remote path (keybindings editing, diagnostics, context menus — rethink with in-app equivalents).
  Evidence: settings-panels.tsx openInPreferredEditor + AboutVersionSection (window.desktopBridge); appearance AppIconSettingsRow (desktopBridge+mac); ArchivedThreadsPanel handleArchivedThreadContextMenu (readLocalApi().contextMenu.show).
- **The search index is a parallel hand-maintained copy of the panels** — settings-preference-index.ts re-declares every setting's id, title, description, section, and search terms separately from the panels that render them, kept in sync by matching preferenceId strings and DOM ids. The jump-and-highlight search UX is worth keeping, but the duplication invites drift (a renamed row silently loses its search entry). Prefer deriving the index from a single panel/registry definition.
  Evidence: settings-preference-index.ts SETTINGS_PREFERENCE_ENTRIES mirrors settings-panels.tsx preferenceId props; settings-search-context.tsx scrollToPreference relies on settingsPreferenceDomId matching.

## shell-chrome

- **Per-workspace panel memory is the shell's backbone — but it's per-device** — Every panel dimension and state (widths, open/collapsed, active tab, tab set, terminal sessions, browser state, secondary rail, tilesets) is keyed by workspaceKey and persisted to localStorage with schema validation and legacy-key migration. This 'your layout comes back per project' behavior is a keep. But it lives only in localStorage, so it is per-device: under ADR 0022 remote parity a phone/browser opens the same workspace with a blank layout. The rewrite must decide explicitly whether shell layout syncs through the Core or stays device-local.
  Evidence: stores/shell-panels-store.ts, stores/workbench-tab-store.ts, components/chat/view/chat-pane-tiling-store.ts all read/write window.localStorage keyed by workspaceKey/routeKey; no server round-trip.
- **The whole shell assumes a mouse; ADR 0022 parity has no touch story** — The most distinctive shell interactions are HTML5-drag or hover only: drag-to-tile from the sidebar with 5-zone drop overlays, drag-to-reorder workbench tabs, dragging 'New Agent' to split, column-resize sashes, and hover-only tab close buttons. None translate to touch. Since ADR 0022 makes a phone/browser client first-class, the rewrite must keep the clean underlying state algebra (split/expand/focus/close, tab move) but give every one of these a non-drag, touch-viable control.
  Evidence: chat-pane-tiling-surface.tsx (dragstart/dragover/drop + getDropOverlayBounds), right-workbench-header.tsx WorkbenchTabClusters drag reorder, app.tsx useColumnResize sashes, tab close button opacity-0→group-hover.
- **Layout already thinks in modes, not pixels — build mobile on that** — useShellPanelModes observes shell width and only re-renders React when a mode enum flips (inline↔overlay), with a debounced exit to stop breakpoint flip-flop, and sidebar/rail already collapse to overlay drawers with a backdrop + Escape. This is the correct seed for phone/browser parity: extend the existing mode system rather than bolt on a separate mobile shell.
  Evidence: components/shell/shell/use-shell-layout.ts (OVERLAY_EXIT_DEBOUNCE_MS, mode-only setState), shell-layout.ts SHELL_BREAKPOINTS/CHAT_NARROW_THRESHOLD, app.tsx ShellOverlayBackdrop.
- **Chrome re-renders, content doesn't — preserve that boundary** — The shell deliberately isolates volatile state (width, fullscreen, drag) into a ShellLayoutService + data-attributes and memoized frames so heavy subtrees (agent sidebar, workbench content) never re-render on resize/fullscreen/drag. This is why maximize and resize stay smooth. The rewrite should keep this 'frame owns layout state, content is stable children' contract as an explicit design rule, not rediscover it.
  Evidence: app.tsx RightAsideFrame/RightWorkbenchContent split with the 'INVARIANT — DO NOT ADD A VOLATILE HOOK HERE' comment; ShellLayoutService imperatively flips data-attrs; LeftAside comment 'No fullscreen subscription.'
- **Post-cutover dead wiring is hiding in the shell and looks alive** — Several shell paths still reference the removed runtime host: the git-agent STOP calls readHonkRuntimeApi().abort() (always rejects 'Runtime host unavailable after core cutover'), agent-runtime-store.markLocalRuntimeThread mutates a store that no longer receives events, and the Dev panel inspects that same dead runtime/session-tree/timeline. These compile and read as working. The rewrite must not port them; stop/interrupt should use the same core interrupt the rest of the app uses.
  Evidence: lib/honk-runtime-api.ts rejectUnavailable() for abort et al; shell-host.tsx stopGitAgentActionMutation → readHonkRuntimeApi().abort(); GAPS.md lists runtimeMode/codexThreadId/replayEvents as legacy-dead.
- **The command palette mixes shipping commands with dev scaffolding** — The palette is the universal entrypoint and mostly excellent (ranked fuzzy search over commands/projects/threads, submenu stack, keyboard affordances). But it interleaves dev-only entries — Component System, HonkKit, Copy server trace, Enable/Disable Multitask flag, Open Dev Panel — gated only by import.meta.env.DEV or a local flag. Separate a real command registry from a dev registry so the product palette stays trustworthy and remotable.
  Evidence: command-palette.tsx pushes DEV-gated 'action:dev:*' items, local-feature-flags 'toggle-multitask-mode-feature', and legacy 'action:open-dev-panel' into the same actionItems array as ship commands.
- **Desktop-only capabilities degrade by hiding/toasting, not by serving** — Add Project toasts 'only available in the desktop app'; git/files rely on the desktop aux server (GAPS.md: absent on web/serve); the browser panel is an Electron webview; open-in-editor and open-keybindings-file need a local editor. ADR 0022 says remote clients get every feature, so the rewrite must give these Core-served equivalents (folder selection, git, filesystem) rather than gate them behind isElectron/desktopBridge presence.
  Evidence: command-palette.tsx openAddProjectFlow guards on window.desktopBridge and toasts otherwise; desktop-api.ts EnvironmentApi git/projects; GAPS.md 'web/serve mode has no aux'.

## sidebar-workspace

- **Organizational state is localStorage-only and won't survive remote parity** — Pins, unread boundaries, status filters, project order, and per-project expand/collapse all persist to a single localStorage blob, keyed by cwd/threadKey. Each client keeps its own copy, so a phone and a desktop on the same account show different pins/unread/order — which cuts against ADR 0022's 'every client gets every feature' intent. The rewrite must consciously split these into per-device (width, maybe collapse) vs account-synced (pins, unread, filters, order), and move the synced ones behind Core. Tellingly, sidebarThreadSortOrder already lives in server-synced client-settings while these don't — the split is currently accidental, not designed.
  Evidence: stores/ui-state-store.ts PERSISTED_STATE_KEY 'honk:ui-state:v1' persistState/readPersistedState (pinnedThreadKeys, projectOrderCwds, sidebarThreadFilters, expanded/collapsed cwds); vs packages/shared/src/client-settings.ts sidebarThreadSortOrder.
- **The sidebar reconstructs row status client-side instead of trusting the Core** — The five-state row vocabulary (idle/running/needs_attention/stopped/error) plus unread is good and worth keeping — but the app derives it by stitching session.activeTurnId, orchestration status, a client visit boundary, and latest-turn fields that Core currently ships as null. This double-sourcing is fragile. The rewrite should have Core serve one authoritative per-row status (including latestReadableAt and needsAttention), and the sidebar should just render it.
  Evidence: components/shell/agents/sidebar/view-model.ts threadState(); use-agent-sidebar-model.ts isUnreadFromVisitBoundary; environments/core/service.ts mapCoreThreadSummary sets latestTurn:null; GAPS.md 'shell rows start with latestTurn: null'.
- **Needs-attention still OR's in approvals, and silently drops plan proposals** — needsSidebarAttention() = hasPendingApprovals || hasPendingUserInput || hasActionableProposedPlan. ADR 0007 removes approvals (Core already hardcodes hasPendingApprovals=false), and ask-question + plan proposals are supposed to survive as Parts — yet Core also hardcodes hasActionableProposedPlan=false, so plan-proposal attention never lights up the sidebar today. Rewrite: define needs-attention as exactly (agent ask-question) OR (actionable plan proposal), and actually wire the plan case.
  Evidence: use-agent-sidebar-model.ts needsSidebarAttention(); environments/core/service.ts mapCoreThreadSummary { hasPendingApprovals:false, hasPendingUserInput:summary.needsAttention, hasActionableProposedPlan:false }.
- **A 'sort order' setting exists that doesn't sort the sidebar** — sidebarThreadSortOrder ('updated_at'/'created_at') is a persisted setting, but the sidebar view-model hardcodes updatedAt-descending within each project group — the setting only affects the command palette and post-archive next-thread navigation. Shipping a named control that doesn't change the thing it names is a UX trap. Rewrite: either make the setting actually reorder the sidebar or remove it. (Separately: there is no sidebar density control at all — conversationDensity is transcript-only — so if the rewrite wants sidebar density it's net-new.)
  Evidence: view-model.ts compareUpdatedAtDesc/sortSidebarChatItems (no sort-order input); consumers only in command-palette.tsx and use-thread-actions.ts; client-settings.ts SidebarThreadSortOrder / ConversationDensity.
- **Grouping is project-first with derived, fuzzy project identity** — Threads group by cwd, labeled from a git-remote repo name or path basename, with heavy machinery (syncProjects, logicalKey vs physical key, worktree paths, projectless threads, 'retained' empty projects) to keep expand/order/pin state stable as cwds remap. This project-first structure is the sidebar's backbone and worth keeping — but the cwd-string keying is where most of the complexity and edge-cases live. Rewrite should keep project-first grouping while having Core serve stable projectId-based grouping so the string heuristics and worktree special-casing can retire.
  Evidence: view-model.ts buildProjectChatSections + shortProjectPathLabel; use-agent-sidebar-model.ts retainedSidebarProjects/projectStateKey; ui-state-store.ts syncProjects logicalKey remapping.
- **Affordances are hover-first, which breaks touch/remote discoverability** — Filter, per-project '+', open-workspace, pin, archive, clear-draft, and the folder/chevron swap are all opacity-0 until hover or focus-within ([@media(hover:hover)] gates), with only a pointer-coarse min-tap-target fallback. On a phone (ADR 0022) none of these are discoverable. The rewrite needs an explicit touch affordance model — e.g., a persistent row overflow/kebab menu — rather than reveal-on-hover.
  Evidence: body.tsx SidebarThreadFilterMenu trigger (opacity-0 … group-hover/workspaces:opacity-65); thread-item.tsx hover action cluster (group-hover/sidebar-item:flex); section.tsx folder/chevron opacity transitions.

## status-notifications

- **Two toast systems ship; one is silently unmounted** — The rewrite must have exactly one notification/toast pipeline. Today base-UI toasts (app/toast.tsx) are mounted and work, while sonner `toast()` is imported in 9 shell files and never rendered (no <Toaster/> anywhere in src) — so 'Path copied', file/save errors, 'Update downloaded', archive/rename failures, etc. produce no visible feedback at all. Pick the base-UI manager and route everything through it.
  Evidence: grep 'Toaster' across packages/*/src returns nothing; sonner imported in shell-host.tsx, git/panel.tsx, files/project-file-tree.tsx, shared/update-pill.tsx, agents/sidebar/{section,thread-item}.tsx, plan/plan-workbench-panel.tsx. sonner CSS survives only in the stale packages/desktop/out build.
- **Thread-scoping defeats cross-thread attention alerts** — Global attention/completion alerts must not be scoped to the active thread. Input-needed toasts are added with data.threadId = the needing thread (taskCompletion.tsx showThreadToast), but the viewport filter only renders a scoped toast when that thread is the active route (shouldRenderThreadScopedToast), and the emitter separately suppresses when the thread IS visible — so a backgrounded thread's 'Input needed' toast is effectively never rendered, and its 'Open' action is unreachable. Model thread-local toasts and global alerts as distinct kinds.
  Evidence: app/toast.tsx:93-116 shouldRenderThreadScopedToast returns activeThreadRef?.threadId === toastThreadId; notifications/taskCompletion.tsx:136-157 shows showThreadToast tags data.threadId; :348-355 suppresses only when visible+foreground.

- **The whole browser surface is Electron-bound and breaks ADR 0022 remote parity** — Every meaningful capability here (the `<webview>` itself, nav state, screenshot, DevTools, storage clearing, localhost scan, CDP automation) reaches through the Electron `<webview>` tag or window.desktopBridge IPC. None of it can run on a phone/browser client. Practical consequence: the 'Agent connected' badge can light up while the agent has no wired way to actually drive the page. The 'Call browser_open first' error string and the cursor-ide-browser MCP references in timeline-render-items.test.ts show the agent path historically came from an external/runtime layer, not the current core.
- **The localhost-preview empty state is the surface's strongest idea and should anchor the rewrite** — The most product-defining behavior is that an empty browser panel auto-discovers the user's running dev server and offers one-click open, with a clear 'Open a local preview' framing. This — not general web browsing — is the reason a coding tool ships a browser: see your app while the agent builds it. Preserve this as the primary flow (and make the agent able to open/inspect that same preview), rather than treating the panel as a generic Chrome tab.
  Evidence: browser-panel.tsx empty state renders 'Open a local preview' with detected localhost buttons; localhostScanState drives 'Looking for localhost servers...' / 'No localhost server detected.'; BROWSER_LOCALHOST_PORT_CANDIDATES targets common dev ports (3000, 5173, 4321, 6006, 8080...) and the placeholder defaults to the first detected server.

  Evidence: browser-location-bar.tsx (showSegmentDisplay swaps sr-only input for segment button), browser-panel.tsx scan loop guards on document.visibilityState and window focus/visibilitychange, right-workbench-header.tsx close button uses opacity-0 ... group-hover:opacity-100 focus-visible:opacity-100.
- **Ship-blocking dead UI: a permanently disabled 'Browser agent' button** — A greyed-out control advertising agent capability with no handler is worse than nothing — it implies a feature that doesn't exist. The rewrite should not carry inert placeholders forward; either design the manual agent-handoff action or omit the button.
  Evidence: browser-subchrome.tsx renders `<BrowserNavButton aria-label="Browser agent" disabled title="Browser agent"><IconFeather/></BrowserNavButton>` with no onClick anywhere.
- **Silent-failure actions leave users guessing** — Screenshot-to-clipboard and storage-clear actions resolve to no-ops (return false / early return) when their bridge APIs are missing or fail, with no toast or confirmation on either success or failure. For a rewrite that must behave identically on remote clients (where these bridges are absent), silent no-ops become invisible broken buttons — actions need explicit availability gating and result feedback.
  Evidence: copyBrowserScreenshotDataUrl returns false silently on any error; takeScreenshot ignores a null dataUrl; clearBrowserPartitionStorage early-returns when window.desktopBridge?.clearBrowserPartitionStorage is undefined — none surface anything to the user.

## workbench-files

- **The entire Files data layer is aux/desktop-only and unimplemented in Core** — Project filesystem browse/read/write/search/rename/delete (and, more loosely, git-status) are the load-bearing dependency of this whole surface, and after the cutover they are notImplemented in the Core environment — the panel is effectively inert against Core today. ADR 0022 remote parity means the rewrite cannot treat the Files panel as 'already works'; folding a project-filesystem plane into Core is a prerequisite, not a detail.   Evidence: packages/app/src/components/shell/files/project-file-editor-shell.tsx:118-199 (save with expectedMtimeMs/expectedSizeBytes, isProjectWriteConflictError, conflict banner at 284-298).

This is exactly the wire model ADR 0013 wants and should be the centerpiece of how the editor talks to the agent in the rewrite — make it prominent (floating widget + keybinding) rather than buried.
  Evidence: project-editor-selection-widget.tsx (Monaco content widget building {path,label,lineStart,lineEnd}); project-file-editor-shell.tsx:205-216 addSelectionToChat -> composerHandle.insertMention/focusAtEnd.
- **Dual editor placement (center vs side-panel) is disproportionate complexity** — Supporting a file in both the right panel and a full-width center surface spawns two mounting components, a persisted placement enum, bespoke empty states ('Return editor to panel' / 'Back to chat'), and a special case forcing the stable Files tab to render empty. It's desktop-shell-shaped with no clean phone analogue. Collapse to a single editor surface model in the rewrite; it removes a whole class of state-sync bugs and the confusing empty-tab rule.
  Evidence: project-center-editor-surface.tsx (parallel toolbar+shell), project-editor-overflow-menu.tsx:74-86 (Open in Center/Side Panel), project-files-panel.tsx:113-130 & 319-342 (stableFilesTabActive / centerEditorActive branching).

The rewrite should keep the lifetime rule but centralize the fs-event -> model-cleanup wiring so it can't drift.
  Evidence: packages/app/src/lib/monaco/project-models.ts (registry, closingKeys, releaseProjectModel keeps dirty alive); markProjectModelClosed called from project-file-tree.tsx (delete/rename) and project-files-panel.tsx / project-center-editor-surface.tsx (close).

They are legitimate desktop shell chrome, but the rewrite (with remote-first parity) should model them as optionally-present capabilities so web/phone builds omit them cleanly rather than rendering dead menu items.
  Evidence: project-file-tree.tsx:277-322 (openProjectFilePath via ensureLocalApi().shell.openInEditor; revealProjectFilePath via window.desktopBridge.showItemInFolder; canRevealInFinder gated by isElectronHost).

## workbench-git

- **The Git panel doesn't do git — the agent does** — The commit surface is delegation, not plumbing. 'Commit & Push', 'Create Branch & Commit', 'Commit & Create PR' all send a scripted natural-language prompt to the project's agent thread (coordinateTurnSend) rather than running git. There is no stage/unstage, no hunk-level staging, no commit-message field anywhere in the panel; the human's role is review (diffs + Viewed checkboxes) plus a one-click 'ask the agent to commit'. The only direct human git mutation is discard. The rewrite should keep this opinionated model but make it legible: today the 'commit' button silently starts an agent turn on some thread, and its pending/stop state is entangled with the chat thread lifecycle, which users won't expect.
  Evidence: packages/app/src/lib/git-agent-actions.ts (prompts: 'stage explicit file paths only, never git add -A', 'Create one concise commit'); shell-host.tsx runs these via coordinateTurnSend/applyLocalThreadTurnStartRequested; the aux git API exposes no stage method and the panel calls only init/discardPaths/getFilePatch/getFileImage/refreshStatus.
- **Review state lives in per-device localStorage — a remote-parity break** — 'Viewed' review progress and the unified/split preference are stored in the browser's localStorage keyed by git root, so they don't travel with the user. Under ADR 0022 (a phone/browser client gets every feature) review progress must be user/thread-scoped in Core, not device-local.
  Evidence: packages/app/src/hooks/use-git-viewed-state.ts (localStorage key honk:git-viewed:<root>); use-environment-git.ts useDiffStylePreference (honk:git-diff-style via useLocalStorage).

The rewrite should only surface wired controls — a menu that shows a toggle or shortcut that does nothing quietly erodes trust in the whole surface.
  Evidence: grep shows panel.tsx ignoreWhitespace never flows into gitPatchQueryOptions; MenuItem 'Find in Diff' has disabled; EMPTY_BRANCH_COMMIT_OPTIONS is always passed as branchCommits; visibleFiles has no 'branch' branch so Branch == working tree.

Any interrupt/stop in the rewrite must go through Core (ADR 0005 delivery modes / honk.threads.interrupt), confirming the general rule that runtime-api-backed features are dead even when the UI looks alive.
  Evidence: shell-host.tsx stopGitAgentAction mutation calls readHonkRuntimeApi().abort({threadId}); lib/honk-runtime-api.ts abort() → rejectUnavailable() ('Runtime host unavailable after core cutover.').

Yet the aux git API already offers listBranches/createBranch/checkout/createWorktree/removeWorktree/pull/resolvePullRequest/preparePullRequestThread (used elsewhere for thread creation, not in this panel). If the rewrite wants a fuller SCM experience for remote clients, the Core capabilities are largely there — the gap is UI, not backend.
  Evidence: packages/shared/src/desktop-api.ts git interface (lines ~404-427) lists those methods; panel.tsx / use-environment-git.ts consume only init, discardPaths, getFilePatch, getFileImage, refreshStatus, onStatus.

This breadth is well-considered and should carry into the rewrite verbatim.
  Evidence: diff-viewer.tsx GitDiffPlaceholder branches; git-image-view.tsx (missing/too_large/unsupported/decode-failed); git-file-type.tsx extension sets; git-diff-card.tsx resolveGitDiffHeaderStatus.

## workbench-plan

- **The same plan is authored in three places — collapse to one** — A single proposed plan is presented and acted on in three parallel surfaces, each with its own Build and its own edit affordance: the inline chat message (ProposedPlanMessage, hover-edit), the composer follow-up tray (Review Plan: Build/View/Dismiss), and the workbench panel (edit/build/copy/download/save). Two of them embed a full Tiptap editor over the same markdown. This is redundancy the user has to reconcile mentally, and it triples the surface area for bugs. The rewrite should pick one authoritative plan surface (the panel) plus one lightweight pointer/prompt (the tray), with a single Build and at most one editor.
  Evidence: proposed-plan-message.tsx (inline edit), chat/composer/plan-follow-up/plan-follow-up-tray.tsx (build/view/dismiss), shell/plan/plan-workbench-panel.tsx (edit/build/export) all render the same activeProposedPlan.
- **Build/completion state is inferred, not reported — and it's fragile** — planBuildStatus is stitched from four signals (a running sourced turn, plan.implementedAt, whether ALL markdown checkboxes parse as [x], and a localStorage 'building' set). 'Built' is decided partly by regex-parsing todo checkboxes out of the plan markdown (arePlanMarkdownTodosComplete). The Core already knows a plan is implemented (implementedAt) and whether its turn is live; the client should read that, not re-derive completion from prose. Rebuild build-status as a single Core-served value.
  Evidence: shell-host.tsx planBuildStatus (583-591) mixes activeProposedPlanTurnRunning, implementedAt, arePlanMarkdownTodosComplete, and buildingPlanKeys; parsePlanMarkdownTodos in plan/proposed-plan.ts.
- **Rich-text (Tiptap) over a plaintext markdown artifact fights ADR 0013** — Plan editing runs the markdown through Tiptap/ProseMirror (StarterKit + Markdown) and needs a syncing dance (syncingRef, lastEmittedRef, normalize round-trips on every keystroke and every external value change) to keep the markdown and the document model in agreement. That is a lot of machinery — and a whole document state — for editing plaintext, and it runs against the raw-text posture of the new wire. A plain markdown textarea or CodeMirror is truer to the artifact and removes the round-trip class of bugs.
  Evidence: shell/plan/editor/tiptap-editor.tsx (setContent/getMarkdown sync effects); tiptap-extensions.ts = [StarterKit, Markdown]; ADR 0013 raw-text wire.
- **Plan lifecycle state lives in localStorage — it won't survive remote parity** — Whether a plan tray is dismissed and whether a plan is 'building' are stored per thread:plan key in localStorage. On a second device the dismissed plan reappears and build state is lost. Any state that changes what the user sees about a plan (dismissed, building, implemented) is thread state and belongs in the Core so every client agrees (ADR 0022). Device-local storage is only appropriate for truly ephemeral view preferences.
  Evidence: stores/proposed-plan-lifecycle-store.ts STORAGE_KEY 'honk:proposed-plan-lifecycle:v1', persistDismissedPlanKeys/buildingPlanKeys.
- **Client-side patching of a server Part is an anti-pattern to retire** — The todo auto-sync effect keeps two representations (the plan's markdown checkboxes and the live todo activity steps) in agreement by rewriting the plan's markdown string on the client and pushing an update command. This couples display state to persisted state through string surgery, and under core the update isn't even served. The lesson for the rewrite: the Core owns plan/todo state as structured data; the client renders it, never edits the server's copy to reflect derived progress.
  Evidence: shell-host.tsx todo-sync effect (708-758) calling syncPlanMarkdownTodosWithSteps then thread.proposed-plan.update; GAPS.md lists 'plan markdown updates' as unsupported by core dispatch.
- **Presenting the plan as an actionable, dismissible prompt at the composer is the pattern to keep** — The one clearly good move here: after a turn settles, the actionable proposed plan surfaces as a compact tray directly above the input with Review / Build / View / Dismiss — decision placed exactly where attention already is, gated so it doesn't fight pending questions. Carry this pattern forward (as the single tray), and keep the small editorial touches that make plans readable (title lifted to chrome, redundant heading/Summary stripped from the body).
  Evidence: plan-follow-up-tray.tsx; chat-view.tsx showPlanFollowUpPrompt gating (980-983); plan/proposed-plan.ts stripDisplayedPlanMarkdown/proposedPlanTitle.

## workbench-terminal

- **Terminal activity is invisible — and the events that would show it don't exist** — The whole running-subprocess story (useWorkbenchTerminalRunning + the confirm-close-running dialog) is built on 'activity' events that Core never emits, so it silently no-ops: the running set is always empty, the destructive-close guard never fires, and there is no tab badge for a busy terminal anywhere. For a terminal meant to run long jobs, 'is something still running here?' is a core question the current design cannot answer. The rewrite must decide deliberately: either Core emits subprocess-activity events (then wire a tab indicator + safe-close guard with real remote parity), or cut the machinery entirely instead of shipping dead code.
  Evidence: packages/app/src/environments/core/GAPS.md ('Core attach does not expose subprocess activity … no terminal activity events'); use-workbench-terminal-running.ts only mutates 'running' on 'activity' events; shell-host.tsx:1396 gates the AlertDialog on runningTerminalIds.has(...), the dialog's only consumer.
- **Clear is cosmetic-only under core/v1** — Ctrl+L / Cmd+K clears the local xterm buffer but the server clear is rejected-and-swallowed, so scrollback is never truncated on the PTY and a reload re-hydrates the full history — the 'cleared' terminal comes back full. A clear that silently doesn't clear is a trust bug. The rewrite should either add a real clear-history verb to Core or make the shortcut honestly visual (scroll-to-bottom / soft clear) and label it as such.
  Evidence: environments/core/terminal.ts clear() returns Promise.reject(CORE_TERMINAL_CLEAR_UNSUPPORTED_ERROR); panel.tsx:225-231 calls api.clear(...).catch(()=>undefined); GAPS.md notes clear/history truncation and deleteHistory are unsupported.
- **Two competing session lists for the same terminals** — The same terminal sessions are rendered twice — as pills in the shared workbench tab strip and as rows in a separate collapsible rail — each with its own activate/close affordances, plus a sub-chrome row whose main job is toggling that rail. This is duplicated navigation and duplicated state to keep in sync. The rewrite should pick one model; tabs are the stronger primitive because they're already shared with files/browser/dev/plan, which would let the rail and its toggle row disappear.
  Evidence: terminal-rail.tsx renders sessions list; workbench-tab-store.ts terminalSessionTab() renders the same sessions as tabs; workbench-subchrome.tsx exists mainly to toggle the rail via shellPanelsActions.toggleSecondaryRail.
- **'Split' is a lie — it just opens another tab** — terminal.split is aliased to terminal.new; there is no pane tiling. A shortcut named 'split' that silently creates a sibling tab trains users to distrust the vocabulary. Either implement real split panes in the rewrite or drop the alias so the surface's affordances mean what they say.
  Evidence: packages/app/src/components/shell/shell/app.tsx:904 — `if (command === 'terminal.new' || command === 'terminal.split') { createTerminal(...) }`.
- **macOS line-editing translation makes the embedded terminal feel native** — Intercepting Alt/Cmd+arrows and Cmd+Backspace and injecting the corresponding control sequences lets the terminal behave like a native Mac text field — a small, high-leverage detail that separates a polished embedded terminal from a raw xterm. Worth keeping and generalizing (cross-platform, more editing shortcuts) in the rewrite.
  Evidence: keybindings.ts terminalNavigationShortcutData / terminalDeleteShortcutData map keys to TERMINAL_WORD_*/TERMINAL_LINE_*/TERMINAL_DELETE_TO_LINE_START; panel.tsx attachCustomKeyEventHandler forwards them via sendTerminalInput.
- **Exited terminals have no recovery affordance** — When a shell exits, the panel ignores the 'exited' event and leaves dead output on screen; there is no restart button, and only closing the LAST terminal implicitly re-spawns (via a resetKey remount). Recovery via 'close then create new' is clumsy for what should be one click. The rewrite should render an explicit exited state with a Restart control (the Core restart verb already exists).
  Evidence: panel.tsx onEvent handles output/cleared/started/restarted but not 'exited'; shell-host.tsx closeTerminalSession only bumps resetKey on the isFinalTerminal path; environments/core/terminal.ts exposes restart but no UI calls it directly.
- **Tab labels guess the shell from process.env — accurate only on desktop** — The terminal tab/caption label comes from inferLoginShellCaption(), which reads process.env.SHELL (Node/Electron only) and otherwise falls back to a 'zsh'/'powershell' guess from the user agent. Under ADR 0022 remote parity, a phone/browser client will mislabel every terminal. The Core should report the real shell so all clients label sessions correctly.
  Evidence: lib/shell-caption.ts reads process.env.SHELL with a navigator.userAgent fallback; used for both the tab label (workbench-tab-store) and the sub-chrome caption (shell-host.tsx:1178).
- **Missing table-stakes terminal ergonomics** — The xterm is created with only the FitAddon — no link detection (URLs in output aren't clickable), no in-terminal search/find, no copy affordance beyond native selection. For a terminal users will paste build logs and URLs into, these absences are felt. The rewrite should add WebLinks + a search addon as baseline.
  Evidence: panel.tsx constructs `new Terminal({...})` and loads only `new FitAddon()`; no SearchAddon/WebLinksAddon/clipboard addon anywhere under components/shell/terminal/.
