# Chat Runtime Timeline Implementation Notes

## Decisions

- Runtime chat rendering is driven by `RuntimeDisplayTimelineProjection`, not by reconstructing legacy `WorkLogEntry` rows in the app.
- Runtime `tool`, `extension-ui-request`, `custom-message`, and `thinking` items are first-class timeline entries. Legacy `work` rows remain only for non-runtime orchestration timelines.
- Pending user sends are emitted by the runtime as typed user `message.completed` events with `clientMessageId`, so first-send rendering is owned by the runtime timeline instead of an app-side transient append.
- ChatView treats a thread as runtime-owned before the first display timeline item exists. Ownership can come from a host display timeline, a host session tree, or a local send marker set before `sendRuntimeTurn`. This prevents the first send from briefly using legacy transient message rendering and then switching to runtime rendering.
- The local runtime ownership marker is cleared only after positive host ownership arrives for the same thread: session tree, display timeline, or pending extension UI. Empty snapshots do not clear it, so a stale startup snapshot cannot reintroduce the first-send legacy flash.
- During runtime startup gaps, ChatView materializes only user message rows that are already local/committed/pending. This applies both before the first display timeline exists and after the host publishes an empty display timeline before the runtime user-message event arrives. It does not materialize assistant placeholders, work rows, or tool rows from the legacy path.
- Pending optimistic rows are acknowledged by canonical message ids, including runtime display timeline user-message `clientMessageId`s. This removes the stale window where runtime had accepted the send but the optimistic row stayed alive until committed server messages caught up.
- Pending acknowledgement must run even when committed server messages are empty; runtime-only `clientMessageId` acknowledgement is the first-send path for a fresh runtime thread.
- Runtime thinking is rendered as its own row with `data-message-kind="thinking"`, matching Cursor's explicit `thinking` discriminant instead of using legacy task activity rows.
- Runtime tool rows expose Cursor-style wrapper attributes: `data-tool-call-id`, `data-tool-status`, and `data-tool-has-error`.
- Multikit owns the shared `tool-call-shimmer` animation and the reusable tool-call row/task/shell slot chrome. App chat rows still own runtime-specific artifact parsing, shell output truncation, file/url callbacks, and subagent transcript composition.
- Cursor's layout primitives are covered by `@multi/multikit/layout` (`Stack`, `Row`, `Grid`, `Spacer`). The API stays structural and does not move app-specific chat/workbench composition into Multikit.
- `/dev/multikit` previews are treated as required primitive coverage. Chat-adjacent previews expose status/truncation/action states, not runtime snapshots or data-loading behavior.
- DialKit preview coverage is component-specific, not every prop for every component. Simple primitives expose their meaningful visual axes; chat-facing primitives get scenario previews for combined states such as toast stacks and tool-call status matrices.
- Multikit now centralizes button-like cursor affordance through shared cursor helpers. App code should not add `cursor-pointer` for normal Multikit controls; listbox/menu option rows intentionally keep `cursor-default`.
- A five-agent council pass completed after the cursor/style cleanup. The council confirmed export/catalog/preview registration is complete, but found remaining app migrations, weaker chat-scenario DialKit previews, and primitive API issues around Workbench buttons, ToolCallLine click semantics, Text typing, ToastAction forwarding, and app-owned task/shell tool CSS.
- `/dev/multikit` preview coverage was strengthened for several council-flagged gaps: `Card` now exercises `CardFooter` and actions, `Table` covers row/column count, empty body, long cells, density, and width, `Alert` covers icon/action/titleless/long-copy cases, `Popover` covers side/align/tooltip-style/instant/workbench variants, and `Menu` covers workbench items, icons, disabled rows, checkbox rows, and radio rows.
- `/dev/multikit` now treats DialKit as universal primitive coverage plus targeted scenario coverage. All 49 exported visual primitives are registered and previewed; chat-facing previews go deeper where combined state matters: `Command` covers palette/composer/positioned-composer plus loading/empty/results, `Sidebar` covers queue-style tray rows, and `Toast` covers stack/collapsed/error-copy/anchored-tooltip scenarios.
- `SidebarTrayRowContent` now owns the shared `overflow-hidden` clipping behavior for tray rows, removing the queue-specific app CSS override from `conversation.css`.
- `@multi/multikit/tool-call` now includes task and shell slot primitives (`ToolCallTaskRoot/Header/Body/...`, `ToolCallShellRoot/Header/Body`, and `ToolCallLineChevron`). `ToolCallRenderer` composes those slots while keeping command artifact lookup, output truncation, approval expansion state, and nested subagent rendering in the app.
- The app `tool-call.css` file was tightened to lifecycle hooks only; task/shell layout and reduced-motion behavior moved into Multikit classes so future chat work has fewer custom selector buckets to copy.
- The `/dev/multikit` `tool-call` preview now covers line, status-matrix, task, and shell scenarios, so the extracted task/shell slots are previewable without a live chat runtime.
- Generic tool-renderer expansion/file controls now use Multikit `Button` while preserving `ToolCallLine` class variants. This removes raw button and `div role="button"` usage from `tool-renderer.tsx` without moving runtime-specific edit/diff/output rendering into Multikit.
- Markdown mermaid controls and code-copy controls now use Multikit `Button` while preserving markdown-specific positioning and reveal behavior.
- Composer controls now use Multikit `Button` across the audited composer folder. The streaming submit/stop controls use an app-owned `ComposerActionButton` wrapper so form submit, interrupt callbacks, running/busy metadata, send-while-streaming labels, and spinner rendering stay in the composer while the underlying control comes from Multikit.
- Composer attachment previews, pending user-input option rows, and inline command chips now use Multikit `Button` with local chrome stripped where the editor or media surface owns layout.
- The audited chat component tree no longer contains raw `<button>` elements. Work-group headers, subagent status rows, image preview/backdrop controls, and subagent tray click-capture controls now use Multikit `Button` with chrome stripped where those surfaces are structural rather than visible button chrome.
- Root route error details now use Multikit `Collapsible` + `Button` instead of native `details/summary`.
- Read-only rich text rendering now uses Multikit `Pre`, `Code`, and `Link` for fallback rich-text code/link surfaces.
- Proposed plan editing now uses Multikit `Textarea`; `Textarea` gained a `controlClassName` hook so app code can tune the inner editing control without replacing primitive field chrome.
- Chat header title and scroll-to-bottom controls now use Multikit `Button`; backdrop/click-capture and image zoom hit targets are also Multikit-backed, but their geometry and behavior remain app-owned because they are structural/media controls rather than normal button chrome.
- `WorkbenchIconButton` and `WorkbenchTextButton` now compose Multikit `Button`, so workbench controls use the same Base UI button foundation and cursor handling as other controls while keeping their existing public API.
- A WorkbenchChromeRow subagent review confirmed the row primitive should own chrome typography and slot geometry, while app call sites should supply only product content. `WorkbenchChromeRow` now sets the row font baseline and exports action group, divider, label, spacer, and text-control variants. Shell/git call sites use those slots instead of repeating row height/gap/divider/text class clusters.
- Cursor's local branch chrome does not add a branch icon or monospace branch label. `LocalBranchBar` now renders the branch as plain WorkbenchChromeRow text with no icon and no `font-mono` override.
- Workbench row CSS hooks such as `ui-tab-system`, `editor-panel-tab-root`, `editor-panel-tab-bar-tab-cluster`, and `editor-panel-tab-bar-spacer` remain centralized in Multikit because they appear to drive tab sizing, drag regions, and existing selector compatibility.
- `ToolCallLine` now uses Base UI `Button` for clickable rows instead of `div role="button"`.
- `Text` keeps Cursor-aligned `as` semantics but now types element-specific props better and no longer emits an unused `data-element`.
- `ToastAction` forwards `className` to custom `render` elements instead of only applying it to the default action button.
- Multikit modal backdrops and tab indicators no longer use `transition-all`; transitions are narrowed to the properties they animate.
- `packages/app/src/app` has no remaining native route-level controls from the audited set (`button`, `input`, `select`, `textarea`, `a`, `pre`, `code`). The root route status card keeps only layout sizing in app code; Multikit owns its card chrome.
- Queue tray shell/header chrome is now Multikit-owned through `SidebarTray`, `SidebarTrayHeader`, and `SidebarTrayHeaderButton`; app queue code only supplies queue-specific state/data attributes and behavior. The primitive uses generic `ui-tray` hooks, not queue-specific class names.
- Queued composer row actions and the edit-banner cancel control use Multikit `Button`, leaving queue behavior and labels in app code.
- Chat dismiss/cancel/close controls that are plain buttons now use Multikit `Button`: thread error dismiss, human-message expand toggle, inline edit cancel, plan tray dismiss, and subagent tray close.
- Context-window meter triggers now use Multikit `Button` as their Base UI `PopoverTrigger` render element.
- Raw buttons that remain in chat are intentionally specialized for editor chips, markdown copy/fullscreen, image preview/backdrop, composer submit/interrupt, user-input option semantics, tool/timeline expansion, or click-capture shells. Those need separate component boundaries before migration.
- Timeline row stability is based on visible typed fields, not runtime `eventIds`. `eventIds` are diagnostic lineage and can grow on every stream event without requiring a React row replacement.
- The runtime host applies normal runtime events to the display timeline incrementally. Full timeline projection remains for snapshots, session-tree commits, and pending extension UI changes.
- Runtime tool updates treat `partialResult` as displayable running output. Completed `result` replaces it when the tool finishes.
- Runtime display tool items carry canonical `command` and `output` fields. `command` is only projected from typed tool args, never from `details`, `summary`, or result text; React renders those fields instead of parsing raw runtime payloads.
- The shared shell tool renderer no longer falls back from missing `command` to `details`. A shell row with no typed command renders `Ran` without a fabricated collapsed command label, while output stays in expandable content.
- `durationMs` is not part of the runtime display timeline contract. It remains in legacy artifact analysis and usage/settings calculations, but the new chat renderer does not need duration plumbing to render typed runtime messages or tools.
- Pi live message events do not expose a message id, so live display projection tracks message lifecycle order per `turnId` and role. The first live message keeps the historical `message:<turnId>:<role>` id; later same-turn lifecycles get stable numeric suffixes.
- Runtime custom messages render through a custom-type registry. Known markdown-shaped custom messages render without leaking `[customType]`; unsupported custom types use an explicitly named unknown fallback.
- Raw runtime events are capped in host snapshots and incoming app snapshots. Live `runtime-event` host events still update thread-store side effects, but they no longer append to the app runtime-store snapshot on every stream frame. Display timeline events are the chat render invalidation boundary.
- Display timeline events are ignored by the app runtime store when they only change diagnostic lineage or raw payload identity. Visible fields such as message text/thinking, tool status/command/output, custom message rendered text, extension UI title/message/status, and proposed-plan text still update the store.
- Incoming host snapshots also preserve existing display timeline object identity when the new timeline is visibly equal. This prevents a full snapshot from undoing the live `display-timeline` no-op behavior and rerendering ChatView with identical rows.
- Pending extension UI request arrays are also preserved when request data is unchanged. ChatView reads active extension requests from this store, so repeated equivalent request lists should not invalidate the active chat.
- App runtime snapshots are normalized before syncing into the thread store, so oversized incoming snapshots do not replay unbounded raw events. Display timelines also count as active runtime-thread evidence during stale-thread cleanup.
- Runtime display projection now treats the synthetic prompt user `message.completed` event as the lifecycle anchor for that `turnId:user` row. Later Pi user lifecycle events for the same turn merge into that row and preserve its `clientMessageId`, so React receives one canonical user item instead of two user bubbles.
- `implementation-chat.html` now includes the runtime-to-UI layer diagram and a GPT-5.5 medium council-refined list of 10 next improvements. The best next targets are typed runtime tool variants, Effect-decoded Pi tool payloads, unified extension UI and approvals, checkpoint/restore rows, first-class task/subagent tools, streaming/tail status separation, ChatView timeline extraction, content-aware row sizing, visible metadata without row churn, and a golden runtime-to-UI fixture matrix.
- The same HTML artifact now includes an implementation matrix for those targets. It identifies the owning layer, the first code move, and the proof needed for typed tool unions, Pi payload decoding, renderer registries, streaming buffers, ChatView extraction, and Cursor-style DOM row contracts.
- Runtime tool display is now the canonical UI shape for shell/read/grep/edit/MCP/unknown tools. Compatibility `command`/`output` fields remain on tool items, but chat row equality ignores them whenever typed `display` exists.
- Runtime display projection now merges tool lifecycle state first, then recomputes typed display from the merged `toolName`, `args`, `result`, `details`, and `output`. There is no separate per-kind display merge switch.
- The app has one `runtimeToolDisplaySignature` helper for visible typed tool equality. Both the runtime store and timeline row stabilizer use it, so visible tool fields are not chosen in two places.
- Runtime subagent tools are now first-class typed display rows. Pi's `SubagentToolDetails` projects to `display.kind = "subagent"`, the chat step renderer passes runtime row context through, and the runtime tool message reuses the existing subagent status/tray surface while mapping fallback rendering to `taskToolCall`.
- Runtime subagent projection now decodes `SubagentToolDetails` with the contract schema instead of casting partially checked arrays. Malformed details fall back to an unknown tool display instead of creating invalid subagent rows.
- Runtime subagent rows now use the existing task tool-call chrome and render the subagent status surface inside it by default. When tray context is unavailable, the task row still shows the subagent completion summary.
- Runtime subagent activities now populate both activity logs and tray transcript items, so child prompt/assistant/reasoning/tool lifecycle activity can appear in the existing subagent tray model.
- Pi keeps its own event and display model. Cursor is used as a presentation reference only: Pi projects `SubagentToolDetails` into a first-class `subagent` display variant, then the app renders that variant with Cursor-style task chrome.
- Runtime subagent task rows now use `Task` as the visible action and the subagent prompt as the subtitle for single-run tasks. The child status row still owns nickname/model/latest result, so the row reads like a task while preserving Pi-specific subagent state.
- Runtime subagent transcript mapping drops the synthetic child `user_message` when its text is identical to the run prompt. The prompt is already visible as the task subtitle, so this avoids a duplicate user bubble in expanded subagent detail surfaces without text-based deduping top-level chat messages.
- Subagent display signatures include all run states but only the last 80 visible activities per subagent thread plus the total activity count. This keeps store and row equality bounded during long streaming runs while still updating on every rendered transcript change, even when parallel children stream at different rates.
- The visible subagent activity cap is exported as `MAX_RUNTIME_SUBAGENT_VISIBLE_ACTIVITIES` and shared by the signature and renderer paths. This keeps React row equality and actual transcript rendering on the same boundary. The signature intentionally excludes the raw full-history activity count; non-rendered or unrelated activity records should not invalidate a visible row.
- Runtime subagent transcript dedupe normalizes prompt whitespace before comparing the child `user_message` with the run prompt. This keeps producer formatting differences from recreating the duplicated user bubble.
- Task rows now resync their expanded state when a default-expanded task body arrives after the header first mounted. This handles the runtime sequence where the tool row appears before subagent details are available.
- Removed the unused `ToolCallTaskTurns` primitive/export from Multikit instead of keeping an uncalled task-body abstraction.
- Runtime tool rows now require a typed `display` payload in the contract. React no longer has a legacy missing-display fallback that guesses command/tool rendering from raw output fields.
- ChatView no longer lets an empty or user-only runtime display timeline replace an existing committed chat timeline. This targets the visible startup flash where the chat could briefly collapse while the runtime display timeline was present but not yet carrying any response/tool rows.
- Runtime user echoes no longer acknowledge pending user rows until the runtime display timeline has a materializable response row. This closes the follow-up flash where a pending user bubble could disappear before the runtime timeline was allowed to render.
- Runtime startup replacement also treats pending/transient rows as preserved content. A user-only runtime timeline cannot remount a pending user row under a different runtime row id before the runtime has a response item.
- Live dev app sanity check: with the existing `Multi.app` dev window at `127.0.0.1:5752`, sending the harmless prompt `please reply ok` produced one stable user row and one `ok` assistant response. No duplicate or disappeared user row was visible at the sampled post-send state. The default `Commit & Push` action was still not clicked because it can mutate the repo.
- Committed timeline work rows are no longer hidden merely because a thread is runtime-owned. They remain available as the startup-gap fallback and are naturally replaced once runtime display rows become usable.
- Runtime display timeline equality now compares required typed `display` data only; the old missing-display command/output equality fallback was removed.
- Multikit task tool-call slots now use compact line-like chrome: inline width, `min-h-6`, baseline title/subtitle, `data-tool-call-line` on the task header, and an indented expanded body. This keeps the subagent tool row closer to Cursor's text-style task tool row without app-level CSS overrides.
- Runtime display materialization now ignores empty live message lifecycle placeholders. A `message.started` with no text, rich text, attachments, or thinking no longer creates an empty assistant/user row, which removes another startup flash path; attachment-only existing user messages still render.
- Runtime subagent rendering maps only the latest 80 activities per subagent thread into logs/transcript items. The display contract still carries the full activity history, but React no longer rebuilds unbounded arrays for long subagent runs on each stream update, parallel subagents cannot evict each other's recent transcript rows, and row/store equality compares the same rendered window.
- Runtime subagent status surfaces use a separate embedded variant when rendered inside task tool-call bodies. Standalone subagent status rows keep their tray padding, while task-embedded rows avoid double horizontal padding and align with the compact Cursor-style task row.
- Cursor's draft-to-thread transition keeps a stable panel/composer shell and retargets only after the next loaded data handle is available. Multi cannot copy Cursor's Solid service graph directly, so the ported invariant is stable display identity: optimistic, live runtime, and committed session-tree user rows now share `message:${clientMessageId}` whenever the runtime supplies a client message id.
- Runtime live assistant/user messages are no longer suppressed merely because a session-tree entry exists for the same `turnId + role`. Suppression is now visible-content-aware, so a blank or non-equivalent committed shell cannot replace a currently renderable streaming row and cause a response flash.
- Runtime user echoes only hide optimistic pending rows in React when the runtime user item itself has renderable text. Blank runtime user lifecycle shells can no longer erase the pending bubble before the replacing source can draw the same user message.
- Pending timeline-row store cleanup is back to committed server messages only. Runtime live acknowledgement can suppress an on-screen duplicate when renderable, but it no longer deletes the local optimistic row from the store before the committed conversation catches up.
- Promoted draft routes now wait for a renderable committed user start before replacing the draft route identity or finalizing draft metadata. `threadHasStarted` is still available for general thread lifecycle and hero-state decisions, but draft-to-thread promotion uses `threadHasRenderableUserStart` so a server session shell cannot tear down the draft view before a user bubble exists.
- Draft routes also wait for `threadHasRenderableUserStart` when a same-id server thread appears before the draft store has an explicit `promotedTo` marker. This closes the recovery/bootstrap path that could otherwise bypass the renderable-content gate.
- The draft canonical route decision is now isolated in `resolveRenderableDraftCanonicalThreadRef` and covered by tests for both explicit `promotedTo` and same-id server fallback paths. This keeps the Cursor-style loaded-content gate out of route JSX condition drift.
- Server-thread routes no longer persist the server route target while a matching draft route exists and the server thread is not yet renderable. During promotion, `multi:last-chat-route` stays pointed at the draft until the committed user start is visible, so a hard refresh cannot bypass the draft shell and reopen an empty server route.
- First-send draft promotion no longer navigates optimistically from the draft route to the server route. The draft session is marked as promoting and pending rows are copied to the server key, but the UI remains draft-rendered until `resolveRenderableDraftCanonicalThreadRef` proves the server thread can draw the same user start.
- Draft routes with a resolved canonical server thread now schedule canonical navigation while continuing to render the draft `ChatView` until the router actually moves. The canonical ref is a navigation target, not permission to swap the still-mounted draft route body to `routeKind="server"`; doing both could produce an extra chat-body teardown during promotion.
- Server-thread routes that are reached during promotion now render the draft `ChatView` shell while `resolveDraftPromotionRouteTarget` returns a draft target. This covers direct route/persistence recovery without letting the server-shaped panel flash empty content.
- Server-thread routes that are waiting for an environment snapshot or redirect now render a stable loading shell instead of returning only sync/null. Cursor has explicit loaded-handle/loading state; this keeps Multi from blanking the chat outlet while route sync catches up.
- Server-thread promotion rendering now uses one stable `ChatThreadMainPanel` instead of swapping `PromotingDraftThreadMainPanel` and `ChatThreadMainPanel`. The panel chooses draft/server `ChatView` props from `resolveDraftPromotionRouteTarget`, keeping the route's primary shell mounted through the handoff.
- Server-thread promotion route reads the matching draft id and draft session through one shallow-compared selector. This removes split subscriptions for the same draft mapping at the draft/server handoff boundary.
- `MessagesTimeline` is no longer keyed by the live `leafId`. Normal thread growth can move the leaf forward every turn, so the timeline cache key is now stable per active thread and the virtualizer is not remounted just because the branch leaf advanced.
- The timeline cache-key invariant is now covered by `chat-view-timeline-cache.test.ts`: changing `leafId` for the same active thread must not change the timeline cache key.
- Reusable virtualizer measurements now skip legacy grouped work rows. Work-group expansion is local UI state, so carrying an expanded measurement into a collapsed row after navigation can mis-size the timeline.
- Thread derivation caches `collectByIds` results by the ids-array and by-id-map references. A new environment state object with unchanged thread slices can now reuse the same derived `Thread`, avoiding active ChatView recomputation from unrelated environment changes.
- Message timeline row identity is now canonicalized with `timelineMessageEntryId(message.id)` for pending, runtime display, and committed chat timeline message entries. Upstream runtime item ids and orchestration row ids still exist as source metadata, but React/virtualizer display identity no longer changes when the same message moves from optimistic to runtime to committed data.
- Cursor's subagent UI uses two render surfaces instead of recursively embedding a full chat: a compact inline task/tool row and a focused subagent preview/tray keyed by child composer/subagent identity. Multi now keeps the same invariant for Pi events: the shared chat projection ignores `subagent.*` child activity rows, while focused subagent transcript projections are owned by the thread store and selected by the tray by subagent id.
- The subagent tray store now stores selection identity only (`subagentThreadId`, `threadId`, `agentId`) instead of live `WorkLogSubagent` objects. This removes the hot signature path that stringified transcript/log text on every child stream delta and removes the `ChatView` relay loop that pushed focused-tray updates back through the shared chat render path.
- Thread activity writes maintain ordered activity slices with targeted upserts instead of sorting all retained activities before storing. Focused subagent projections are still recomputed from the bounded activity slice when that slice changes, but the tray no longer performs that derivation during render.
- Focused tray transcript rows are memoized with field-level transcript item equality. The selected subagent projection can still rebuild bounded transcript objects during a stream tick, but unchanged rows should not re-enter `StepRenderer`; only the active streaming row should redraw.
- Focused subagent projection is now store-owned as `EnvironmentState.subagentByThreadId[threadId][subagentThreadId]`. The tray no longer reconstructs activity arrays or calls session derivation helpers during render; it only selects the projected `WorkLogSubagent`.
- Runtime subagent activity ids are now stable by child thread and item identity (`thread`, `state`, or `item:<itemId>`). Child stream updates replace the existing activity row in the subagent tool details instead of appending a new numeric sequence row, matching Cursor's update-existing-task-card model and preventing the parent tray/display history from growing with every token update.
- Subagent lifecycle item updates are treated as typed snapshots. Only `subagent.content.delta` appends stream text; repeated `subagent.item.updated` rows replace the existing transcript item text so full-message snapshots do not duplicate assistant output.
- Live subagent tool updates now write to a separate subagent activity backing store (`subagentActivityIdsByThreadId` / `subagentActivityByThreadId`) and update `subagentByThreadId` from there. Parent `Thread.activities` and `chatTimelineRows` no longer change for child transcript/status streaming, matching Cursor's rule that nested subagent content does not become top-level chat state.
- Runtime display row equality now ignores subagent activity details. Parent runtime tool rows still update when run-level fields change (`state`, `finalText`, `errorMessage`), but child transcript/activity churn is tray-owned and does not invalidate the parent timeline row.
- Orchestration `thread.activity-appended` events with `subagent.*` activity kinds route directly to the subagent activity backing store. They do not mutate parent activities or parent `updatedAt`, so shell/thread selectors are not invalidated by child-only streaming.

## Tradeoffs

- A GPT-5.5 medium detail-audit council produced 10 concrete follow-ups focused on Cursor comparison, macOS Retina text crispness, typography token consistency, and UI density. Priority order: remove negative chat markdown tracking; migrate git custom popups to Multikit Menu; tokenize markdown headings, workbench menu typography, autocomplete typography, Git commit split button, Git diff-card header controls, and selected-diff scrolling motion; then reuse shared tool-call chevron motion and disable toast spinner animation under reduced motion.
- Runtime tool row stability uses visible typed fields only. Raw `args`, `result`, and `details` payload identity is ignored because those values are diagnostics/source data, not the rendered collapsed row contract.
- For typed runtime tools, stale compatibility `command`/`output` changes are diagnostic churn. The renderer reads `display`; legacy fields only participate in equality when `display` is absent.
- MCP display no longer repeats the parent `toolName`. The display variant only carries MCP-specific provider identity; the surrounding tool item remains the source of the tool name.
- The subagent display kind is Pi-facing rather than Cursor-facing. Cursor's `taskToolCall` remains the styling target in the renderer, but the runtime contract names the data after Pi's actual subagent tool and details schema.
- Completed runtime subagent rows prefer child `finalText`; failed or aborted rows prefer `errorMessage`. Generic activity labels are only fallback status text.
- Synthetic live runtime messages use structural equality for the fields rendered by chat rows. This keeps virtualizer row identities stable when the runtime projection emits fresh objects with unchanged visible content.
- Incremental display timeline merging preserves the first event's `createdAt` and `orderKey` for each row, so running/completed updates do not move rows around.
- Generic runtime `tool.*` events no longer create legacy thread activities. Subagent-specific tool payloads still update subagent side effects because tray/focus UI can depend on them.
- Runtime `message.*` events no longer create legacy live assistant messages or thinking activities. The display timeline now owns live user, assistant, and thinking rows.
- Committed session-tree projection still creates legacy completed thinking/tool activities for non-runtime surfaces that read `thread.activities`. Live runtime events do not use that path.
- Live raw runtime events are no longer visible in the app runtime-store snapshot until the next host snapshot. That makes usage/settings history slightly less live, but avoids a global Zustand snapshot change for every streaming frame.
- Diagnostic-only display timeline updates are also not stored live. This can defer updated `eventIds`, raw `args`, raw `result`, and raw `details` until the next full snapshot, but prevents ChatView from rerendering on payload churn that does not change visible rows.
- Snapshot normalization still refreshes raw event history and non-timeline snapshot fields. The identity preservation is scoped to display timelines so chat subscriptions do not rerender solely because the host sent a fresh copy of equal timeline data.
- Pending extension UI equality includes the request fields used by the interactive surface: id, thread/runtime session, kind, title, message, placeholder, timeout, creation time, and options.
- Custom message equality follows the renderer: string content and text entries in `content[]` are visible; `details` is diagnostic. Extension UI timeline rows ignore `value` and `options` because the row renders status/title/message, while the interactive pending request surface reads the separate pending-extension UI store.
- The local runtime ownership marker is intentionally small: it only decides which chat timeline renderer owns the thread during startup. If `sendRuntimeTurn` fails before the host takes over, the marker is cleared with the failed pending row; once the host owns the runtime, the marker is pruned so old local ownership cannot outlive the host runtime.
- The runtime startup user materializer is intentionally narrower than legacy transient entry append. It preserves the user bubble while avoiding the old path that could flash unrelated assistant/work content, and it stops adding the pending row once the runtime display timeline acknowledges the `clientMessageId`.
- Runtime acknowledgement now removes matching pending rows once the runtime user row is renderable. Blank runtime lifecycle shells still cannot delete a pending row, but after a text-bearing `clientMessageId` appears the runtime display timeline becomes authoritative. This avoids stale optimistic rows resurfacing if runtime and committed session-tree updates briefly reorder.
- Local dispatch state now lives in the pending-send store instead of `ChatView` component state. Draft-to-server promotion copies the same dispatch marker with the pending user row, so a route remount cannot drop the send-busy bridge by itself.
- A running session no longer clears local dispatch. The local marker is cleared by terminal states such as settled turns, pending approval/input, error, or later session updates, which avoids losing the bridge during the small window between user acknowledgement and the first response/tool row.
- User-only runtime display timelines are treated as waiting for the first response while the turn is unsettled. This keeps the visual contract Pi-native: the runtime display projection remains authoritative, but a user echo alone is not a rendered response.
- Runtime user acknowledgements no longer remove pending rows from the message identity source. Pending user rows remain available to `timelineMessages` until committed server messages arrive; runtime materialization can then reuse the optimistic `ChatMessage` object instead of synthesizing a new user bubble.
- Runtime user acknowledgements still suppress duplicate transient pending entries in the rendered fallback timeline. This separates identity preservation from duplicate-row suppression.
- User message row stability now ignores non-visible committed metadata (`turnId`, `completedAt`, `streaming`) when the visible user bubble content is unchanged. This reduces first-send bubble re-render when the committed server row replaces the optimistic row.
- Committed assistant session-tree text now covers a live streamed fragment when the committed text extends that fragment. This fixes the `Hi. What do you want to work on` / `Hi. What do you want to work on?` duplicate without hiding a live row when the committed text is genuinely different or stale.
- Normal composer send now separates server turn-start acknowledgement from runtime send success. A runtime failure after the user row is persisted should clear local runtime ownership and dispatch state instead of leaving the view in a waiting runtime-owned state.
- Runtime turn-start failure reporting is now a narrow client command (`thread.turn.start.failed`) that maps to the existing `runtime.turn.start.failed` activity. This preserves the server projection's existing pending-turn cleanup without exposing arbitrary client-side activity appends.
- Runtime availability preflight is named `assertRuntimeHostAvailable` and is deliberately transport-neutral. Chat send paths call it before appending the optimistic row or dispatching `thread.turn.start`, so an absent runtime host keeps the draft/composer surface instead of creating a persisted pending turn.
- The previous desktop-specific helper name was removed entirely. The API is not a desktop-runtime helper; it is the UI/runtime boundary assertion used before any user-visible turn start work.
- Inline edit resubmission now sends the edited turn to Pi runtime after the server accepts the branched `thread.turn.start`. Previously that path could persist a replacement user row and never start runtime work.
- Client-message ids now participate in runtime display item ids for user messages. This is a producer-side identity change rather than an app-only normalization because React, Zustand equality, and the virtualizer all benefit from receiving one canonical row id.
- Content-aware live-message suppression can temporarily show both a live assistant row and a non-equivalent committed assistant row if the host publishes conflicting visible text for the same turn. That is preferable to hiding the only up-to-date streaming content behind an incomplete committed shell.
- The route promotion gate is intentionally user-message-based, not session-based. This may keep a promoted draft route alive for a few more frames while server data hydrates, but that matches Cursor's loaded-handle behavior better than switching as soon as an empty session object appears.
- Canonical draft navigation now happens without an in-route body retarget. This trades away one eager server render inside `/draft/:draftId` to avoid a second timeline remount before the canonical server route owns the view.
- User-message identity is kept out of React text matching. The duplicate first-send fix is lifecycle-based (`turnId + role + clientMessageId`) in the runtime projection, so repeated identical text remains distinct when it arrives as a separate lifecycle.
- Tool streaming treats `tool.updated` `partialResult` as a delta and `tool.completed` `result` as authoritative full output. This keeps partial stream chunks from replacing prior output while avoiding duplicated final output when the complete result arrives.
- Shell command extraction reads the command from `args`, event data, `details`, `result`, and `result.details`. This covers Pi/Cursor-like payloads where the command is not under `args.command` and prevents output/status text from becoming the collapsed `Ran ...` detail.
- MCP runtime tool rows now use Cursor-style action text (`Running MCP` / `Ran MCP`) instead of the generic shell/action verbs.
- Canonical message display row ids intentionally ignore orchestration timeline row ids for `kind: "message"`. That trades away row-id uniqueness as a UI key source for messages, but the message id is the actual stable display object and prevents the same bubble from remounting when the timeline row arrives later.
- Runtime subagent tray history is intentionally recent-tail UI per subagent. Older activity remains in the raw runtime display payload, but the chat row/tray mapper caps rendered logs and transcript items to protect streaming responsiveness.
- Legacy chat timeline derivation remains for non-runtime threads until those surfaces are migrated or removed.
- A follow-up subagent audit reused an existing agent because the local agent thread limit was reached. The audit confirmed the architecture should stay Pi-native and recommended task-row text, synthetic prompt dedupe, and bounded subagent signatures; those recommendations were applied.
- The shared cursor helpers add one small abstraction, but keep the product cursor token from being repeated in every primitive. Badge uses a host-selector helper so passive badges remain neutral while button/anchor badges get the interactive cursor.
- Exact Cursor-name aliases (`IconButton`, `TextInput`, `TextArea`, `Callout`, `Divider`, `Pill`, `H1/H2/H3`) are still intentionally not exported. Current coverage uses Multi names (`Button` icon sizes, `Input`, `Textarea`, `Alert`, `Separator`, `Badge`, `Text as=...`) to keep the API small.

Existing exported primitives have DialKit previews, with the chat-facing ones carrying scenario fixtures instead of only isolated props. Composer submit/interrupt behavior is app-owned and now Multikit-backed through a local wrapper, so it is not a new Multikit export.

- Workbench chrome text intentionally uses native elements and Multikit workbench slots/variants rather than `<Text>`. This follows the repo rule that product surfaces use typography utilities, while `<Text>` is reserved for settings/structured copy.
- Parent runtime display timeline equality now uses the parent tool-display signature. Subagent run state, final text, and errors still update the parent row; subagent transcript/log activity is focused-tray data and no longer replaces the parent runtime timeline.
- Server detail snapshots now reconcile parent-visible thread slices before writing the Zustand store. Fresh mapped arrays from the server reuse the existing message, entry, activity, proposed-plan, turn-diff, and chat-timeline references when their visible content is unchanged, so hidden subagent rows do not recreate the active `Thread`.
- Thread shell equality now compares model selection structurally. The server mapper creates a fresh model-selection object per snapshot, so identity comparison was enough to invalidate the active thread even when the shell did not change.
- Subagent `thread.activity-appended` projection rows are still stored in the activity projection, but they no longer bump the parent thread `updatedAt` or refresh the parent shell summary. This keeps child-agent streaming out of parent sidebar/chat invalidation.
- The snapshot reconciliation uses structural checks only at the server snapshot boundary. Live event paths still use direct upsert logic, which keeps the common streaming path flat while preventing server JSON remapping from looking like parent UI changes.
- Runtime subagent runs no longer publish `finalText` during child `message.updated` or child `message.completed` events. The child transcript still streams through subagent activities for the tray, while parent run `finalText` is assigned only when the subagent run itself completes, so the parent runtime timeline changes at run-completion granularity instead of transcript-token granularity.
- Subagent detail projection now lives in a narrow `useSubagentActivityStore` instead of `EnvironmentState`. Runtime `tool.updated` child activity and server detail snapshots still feed the focused tray, but child-only changes no longer replace the main thread store or wake `ChatView`/shell selectors.

- Focused tray selectors rely on per-thread projection object identity. A regression now covers another thread's subagent stream updating the narrow store without replacing the active thread's projection, so an open tray should not redraw for unrelated child-agent traffic.
- Subagent activity projection now updates only the affected `subagentThreadId`s after a child stream activity changes. The narrow store still keeps ordered activity history for detail reconstruction, but it no longer calls `deriveWorkLogSubagentsFromOrderedActivities` across every child subagent on each token. Snapshot-style `subagent.item.updated` and append-style `subagent.content.delta` regressions preserve sibling subagent object identity.

- The runtime orchestration UI coalescer is part of the subagent streaming boundary. It now has focused coverage proving repeated `subagent.content.delta` and `subagent.item.updated` events for the same child item merge into one UI event before store application, while other child items stay distinct.
