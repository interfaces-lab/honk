Agent chat, from AI event to visible row

The current design has one important rule. Runtime-owned threads should render from a typed display timeline. React should not infer tool rows, user rows, or extension UI rows from raw events if the runtime already projected them.

The live path

This is the path for a normal agent turn after the composer calls sendRuntimeTurn.

Composer send chat-view.tsx creates a client message id and a pending user row.
Runtime host desktop-runtime-host.ts routes the turn to ThreadAgentRuntime.
Pi session thread-agent-runtime.ts emits normalized AgentRuntimeEvent records.
Display projection display-timeline-projection.ts converts runtime events and session entries into typed timeline items.
React chat ChatView materializes entries, stabilizes rows, and renders message components.

Current layers

The names below are the contracts that matter. If a bug crosses one of these boundaries, fix the owner of that boundary instead of adding another UI branch.

1. orchestration

User input Composer text, rich text, attachments, interaction mode, plan follow-up state.
Optimistic row PendingTimelineRow keeps the user bubble visible before runtime acknowledgement.
Send key clientMessageId is the id that joins optimistic UI to runtime projection.

2. runtime

Session events Pi events become message.*, tool.*, extension-ui.*, and plan events.
Session tree Committed Pi entries become durable message and custom-message items.
Display timeline RuntimeDisplayTimelineProjection is the canonical runtime chat payload.

3. app state

Runtime store agent-runtime-store.ts stores display timelines and preserves equal visible rows.
Thread store Legacy messages and activities still exist for non-runtime paths.
Branch filter filterMessagesToBranch and pending-row filtering choose the active path.

4. rendering

Materialize entries runtime-display-timeline.ts maps runtime items to TimelineEntry.
Stable rows timeline-rows.ts reuses rows when visible fields did not change.
Message components Human, assistant, custom-message, thinking, runtime tool, and extension UI rows render here.

Why the first user bubble duplicated

This bug was a layer ownership problem. React was showing what it was given. The runtime projection could produce two user items for the same send.

Synthetic prompt event message.completed with clientMessageId made one user row.
Pi user lifecycle Later message.started or message.completed for the same turn opened another row.
React materialization Two runtime message items became two user bubbles.
Fix turnId + role + clientMessageId keeps that send as one lifecycle.
Result Repeated text in separate turns still renders as separate messages.

What not to do

Do not dedupe user bubbles by visible text in React. Two "hi" messages can be two real turns. The projection must keep runtime identity correct before the UI sees it.

Where rerenders should stop

The chat stream should update when visible content changes, not when diagnostic payloads or raw event arrays churn.

Runtime host

Incrementally applies display-timeline events and refreshes snapshots only when it needs a full projection.

Runtime store

Preserves display timeline object identity when visible fields are equal. Diagnostic fields such as eventIds should not force chat rows to churn.

Timeline rows

computeStableMessagesTimelineRows reuses rows when text, status, command, output, thinking, and extension UI labels have not changed.

Ten improvements to target next

These are the next useful cuts if we want the runtime chat to behave more like Cursor while keeping Pi's typed extension model. The common theme is simple: move meaning down, keep React dumb, and make every row identity explicit.

Typed tool variants in the display timeline

Cursor converts generic tool blocks into cases such as shellToolCall, readToolCall, grepToolCall, editToolCall, and taskToolCall. Multi still carries toolName, args, result, and details as mostly unknown runtime payloads.

Add a discriminated RuntimeDisplayTimelineToolItem.tool union for shell, read, grep, edit, write, subagent, and MCP. Keep the generic case only for unknown tools.

Effect schemas and first-class file artifacts

The contract validates the outer timeline item, but the most important payloads are still Schema.Unknown. That leaves renderers guessing at command, output, file paths, grep matches, and edit results.

Decode Pi tool payloads with Effect schemas at the runtime boundary. A shell row should have typed args.command, exit code, stdout, and stderr. A grep row should have pattern, path, and matches. Read and edit rows should carry file artifacts that React can render without parsing raw result text.

RuntimeDisplayTimelineToolItem.args: Schema.Unknown RuntimeDisplayTimelineToolItem.result: Schema.Unknown Cursor: readToolCall/grepToolCall/editToolCall

Mirror Pi's renderer registries

Pi tools expose renderCall, renderResult, toolCallId, argsComplete, executionStarted, isPartial, and per-row renderer state. Pi also uses registerMessageRenderer(customType, renderer) for custom messages. Multi has many of the tool fields, but React receives them as display data, and custom-message rendering is a hardcoded map.

Create a React-side runtime tool renderer context with the same lifecycle names. That lets built-in tools, extension tools, and unknown tools share one render contract while keeping toolCallId as the component key. Move custom message rendering into a registry module keyed by customType.

Pi: renderCall/renderResult Pi: registerMessageRenderer Pi: ToolRenderContext.toolCallId packages/app/src/components/chat/timeline/step-renderer.tsx

Unify extension UI and approvals in the runtime timeline

Runtime display timeline now has extension-ui-request, while pending extension UI still also syncs through legacy thread-store activities. Approvals still have a separate legacy path. Cursor keeps approval UI in the transcript instead of hiding them behind a side channel.

Put request status, options, current value, timeout, and response affordance in one runtime-owned display item. The app should render from that item and call respondToExtensionUiRequest. Approval rows should follow the same pending, resolved, failed shape.

Add checkpoint and restore rows

Cursor exposes checkpoint and restore actions around agent work. The current runtime display contract has messages, tools, extension UI, custom messages, and plans, but no checkpoint row type.

Add a runtime checkpoint item with restore availability, disabled reason, and restore status. That gives the transcript a durable place for rollback actions instead of burying them in shell or workbench state.

Make subagent and task tools first-class

Cursor has taskToolCall. Multi has subagent payload schemas and tray rendering, but runtime chat still relies on side effects and legacy details for nested agent work.

Add a runtime task/subagent tool case with runs, activities, model, prompt, state, final text, and error. Keep nested transcript churn out of the main row unless the user expands it.

Separate streaming text, turn tail status, and cancel state

The projection updates message text and thinking text directly, and the chat view still has a generic working row. Under load, token updates and tail status can fight for attention.

Keep stable row identity in the display timeline and send text deltas through a small streaming buffer keyed by message item id. Add a turn-tail item for working, cancelling, cancelled, and failed states so cancel UI is not inferred from unrelated rows.

Extract ChatView timeline assembly and row sizing

ChatView owns branch filtering, pending rows, runtime acknowledgement, attachment handoff, runtime materialization, legacy materialization, and edit state. Runtime tools and extension UI also use fixed virtualizer estimates even when their content is much larger.

Move timeline assembly into a hook such as useChatDisplayTimeline. Add content-aware row estimates for runtime thinking, tool, and extension rows so streaming does not cause scroll jumps.

Add visible metadata without causing row churn

Cursor rows carry rich state. Multi has model, token, and subagent metadata in some legacy paths, but the runtime display rows do not carry usage or model fields.

Add optional model, token, and cost fields where the UI actually renders them. Keep them out of visible equality until they appear on screen, so diagnostics do not recreate rows.
